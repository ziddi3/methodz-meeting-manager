# Methodz Meeting Manager v1.1 Release Notes

## Release Theme

Version 1.1 hardens record lifecycle and external sharing without changing the app's offline-first deployment model.

The release adds:

- record retention review metadata
- preservation / legal-hold controls
- a cross-record retention dashboard
- non-destructive partner-safe and public-summary exports
- redaction manifests
- package integrity digests
- external-export activity logging
- archive detail for retention and preservation state

The application remains plain HTML, CSS, and JavaScript. It has no runtime package dependency and continues to work by opening `meeting.html` directly.

## Retention Metadata

Each meeting record can now store:

```text
policyId
reviewDate
lifecycleStatus
note
legalHold.active
legalHold.reason
legalHold.placedBy
legalHold.placedAt
legalHold.releasedBy
legalHold.releasedAt
legalHold.releaseNote
holdHistory[]
updatedAt
```

The default retention presets are operational helpers. They are not legal advice and do not replace review of applicable tax, employment, insurance, litigation, privacy, safety, or contractual requirements.

## Preservation Hold

When a hold is placed, the record captures the actor, reason, timestamp, and a history entry.

When a hold is released, the release actor, timestamp, reason, and a second history entry are preserved.

An archived record with an active hold cannot be permanently deleted through the Archive Vault interface.

The hold is a local workflow safeguard. A future authenticated provider must enforce hold and disposition rules server-side.

## Retention Review Dashboard

The dashboard indexes active and archived records and reports:

- total records
- active legal holds
- records whose review date is due
- records missing a review date

Filters are available for all records, legal holds, due records, and missing dates. The filtered result can be exported as JSON.

## Partner-Safe Export

The external-copy workflow never changes the controlled source record.

Available profiles:

### Partner Safe

Keeps operational meeting content while removing:

- typed signatures
- signature consent and verification metadata
- discussion notes
- protected governance notes
- allowed-role and protected-field configuration
- contact details
- attachment file locations and added-by details
- directory snapshots
- adapter, migration, synchronization, and internal audit metadata

### Public Summary

Exports only high-level meeting metadata, organizations, completed agenda items, approved or confirmed decisions, and the meeting summary.

### Custom External Copy

Allows the user to choose whether to include attendee names and roles, agenda, discussion notes, decisions, tasks, attachment metadata, and a limited retention summary.

Typed signatures and signature-verification fields remain excluded from every profile.

## Redaction Manifest

Every external package includes a manifest containing:

- profile ID and label
- generation time
- source meeting reference
- removed field paths
- warnings
- an irreversible-redaction marker
- `signatureDataIncluded: false`

The package is a separate copy and cannot reconstruct removed fields.

## Integrity Digest

On HTTPS or localhost, the app uses Web Crypto to calculate a SHA-256 digest over a stable JSON representation of the package.

When Web Crypto is unavailable, including some direct-file browser environments, the app uses an explicitly labeled FNV-1a compatibility checksum.

A digest detects package changes. It is not a digital signature, does not prove identity, and does not prove who created or approved an export.

## External Export Activity Log

Completed JSON and HTML downloads add a browser-local log entry containing:

- export timestamp
- format
- redaction profile
- meeting reference
- source type
- source record ID
- integrity algorithm and digest

The log stores export metadata, not a second copy of the meeting content. It can be exported as JSON.

## Schema

The current schema version is `1.1.0`.

`migrations-v11.js` adds retention and redaction metadata while preserving unknown fields and existing record content.

## Deployment

No build step is required.

Supported operation remains:

- direct `file:` use
- localhost
- GitHub Pages
- Cloudflare Pages
- Netlify
- Vercel static hosting
- Render static hosting
- any ordinary static web server

The service worker caches application files only. It does not cache meeting records or attachments.
