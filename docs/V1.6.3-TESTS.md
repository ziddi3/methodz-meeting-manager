# Methodz Meeting Manager v1.6.3 Provider Conformance Tests

## Purpose

This release gate verifies the client contract that future hosted record providers must satisfy without connecting a backend or touching active browser-local meeting records.

The same reusable suite runs against:

- `InMemoryHostedProvider`, which is disposable and process-local;
- `LocalStorageHostedProvider`, bound to a fresh in-memory `Storage` implementation with isolated keys.

The record schema remains `1.6.0`. The app shell remains plain static HTML, CSS, and JavaScript.

## Automated command

From the repository root:

```bash
node tests/v163-provider-core.mjs
```

No package installation is required for this Node test.

## Automated coverage

The conformance suite verifies:

1. required Promise-returning provider operations and contract version;
2. health-check identity, version, and collection metadata;
3. empty-list, missing-record, and `NOT_FOUND` behavior;
4. idempotent create replay;
5. idempotency-key reuse rejection when request content differs;
6. deterministic expected-token conflict handling;
7. provider-version advancement after an accepted update;
8. unknown-field, attachment-reference, recovery, and integrity-metadata preservation;
9. returned-value isolation so callers cannot mutate provider state accidentally;
10. revision snapshot preservation;
11. Archive Vault separation, archived-update blocking, and restore behavior;
12. explicit retryability and partial-failure metadata;
13. duplicate IDs across active and archived collections failing closed;
14. provider export completeness and compatibility-integrity verification;
15. export tamper detection;
16. private JWK and private-key-field rejection;
17. embedded binary and `data:` payload rejection;
18. provider credential-field rejection;
19. explicit permanent-delete intent;
20. cleanup after permanent deletion.

Additional direct tests verify malformed local provider state and a synthetic partial local-storage write.

## CI isolation

`.github/workflows/provider-conformance.yml` runs separately from Playwright browser suites. It checks:

- static v1.6.3 asset wiring;
- app-shell and schema version declarations;
- JavaScript syntax;
- the complete reusable provider conformance test.

This isolation makes provider failures distinguishable from browser UI regressions.

## Manual browser checks

Open `meeting.html` directly and on localhost.

Confirm:

- the main meeting workspace still initializes;
- existing records remain visible;
- draft, archive, revision, retention, redaction, approval, signing, verification, custody, and recovery panels still initialize;
- `window.MethodzHostedProviderContract.version` reports `1.0.0`;
- `window.MethodzHostedProviderAdapters.createDefaultLocalProvider()` returns a valid provider;
- the provider health check reports browser-local collection counts;
- no conformance test writes occur during ordinary app startup;
- direct-file mode makes no network request for records.

### Reference-only attachments

Confirm ordinary attachment references such as an HTTPS file location can round-trip through a disposable provider. Confirm embedded `data:` content and fields such as `contentBase64` fail with `BINARY_PAYLOAD_REJECTED`.

## Failure interpretation

### Contract-shape failure

A provider omitted a required Promise-based method or published the wrong contract version.

### Conflict or duplicate-state failure

The provider allowed a stale or missing expected conflict token to overwrite a record, or permitted the same record ID in active and archived collections.

### Idempotency failure

The provider duplicated an accepted write or allowed the same idempotency key to represent different input.

### Archive or revision failure

The provider lost Archive Vault state, allowed active and archived duplicate IDs, or discarded revision snapshots.

### Unsafe payload failure

A provider accepted private JWK material, embedded binary content, a `data:` payload, or provider credentials. Treat this as a release blocker.

### Integrity failure

The exported canonical payload no longer matches its integrity value. Treat this as a release blocker.

### Partial-failure metadata failure

The provider hid partial success or marked a non-retryable error as retryable without a documented reason.

## Hosted-provider acceptance evidence

Before connecting a real provider, preserve evidence showing:

- the exact provider and contract versions;
- the conformance output;
- server-side authentication and authorization tests;
- tenant-isolation tests;
- encryption and credential-handling review;
- retry and idempotency behavior under network interruption;
- server-side archive, retention, hold, disposition, and recovery behavior;
- proof that private signing keys never enter payloads, logs, backups, or fixtures;
- browser-local fallback and export behavior.

Passing this suite establishes client-contract compatibility only. It does not establish production security, durability, delivery, identity, authority, or legal compliance.
