# Methodz Meeting Manager v1.6.3 Provider Test Plan

## Automated conformance gate

Run:

```bash
node tests/v163-provider-conformance.mjs
```

The test creates only disposable providers. It does not load or mutate the browser’s production `methodzMeetingRecords`, Archive Vault, revision history, or recovery collections.

Both reference providers must pass the identical suite:

1. portable contract shape and version;
2. health check metadata;
3. empty-state behavior;
4. initial record creation;
5. unknown-field preservation;
6. idempotent replay;
7. idempotency-key misuse rejection;
8. duplicate-ID conflict behavior;
9. expected conflict-token update;
10. revision preservation;
11. archive behavior;
12. archived-record update rejection;
13. restore behavior;
14. retryable partial-failure metadata;
15. provider recovery after a synthetic transient failure;
16. export of active records, archives, revisions, and integrity metadata;
17. tamper detection;
18. private-key export rejection;
19. explicit permanent-delete guard;
20. cleanup and not-found behavior.

## CI isolation

The GitHub Actions workflow runs provider conformance in a dedicated `provider-conformance` job. Browser matrix jobs do not depend on it, so a provider-contract failure is distinguishable from a Playwright or browser-engine regression.

The existing static validation job still checks syntax and required file wiring.

## Manual browser checks

Open `meeting.html` directly and on localhost.

Confirm:

- the main meeting workspace still initializes;
- existing records remain visible;
- draft, archive, revision, retention, redaction, approval, signing, verification, custody, and recovery panels still initialize;
- `window.MethodzHostedProviderContract.version` reports `1.0.0`;
- `window.MethodzHostedProviderAdapters.createDefaultLocalProvider()` returns a valid provider;
- the provider health check reports the browser-local collection counts;
- no hosted-provider test writes occur during ordinary app startup;
- direct-file mode makes no network request for records.

## Failure interpretation

### Contract-shape failure

A provider omitted a required Promise-based method or published the wrong contract version.

### Conflict failure

The provider allowed a stale or missing expected conflict token to overwrite a record.

### Idempotency failure

The provider duplicated an accepted write or allowed the same idempotency key to represent different input.

### Archive or revision failure

The provider lost Archive Vault state, allowed active and archived duplicate IDs, or discarded revision snapshots.

### Private-key failure

An export contained an EC private `d` parameter or a recognized private-key field. Treat this as a release blocker.

### Integrity failure

The exported canonical payload no longer matches its integrity value. Treat this as a release blocker.

### Partial-failure metadata failure

The provider hid partial success or marked a non-retryable error as retryable without a documented reason.

## Hosted-provider acceptance evidence

Before connecting a real provider, preserve evidence showing:

- the exact provider and contract versions;
- the conformance output;
- server-side authentication and authorization tests;
- tenant-isolation tests;
- encryption and credential-handling review;
- retry and idempotency behavior under network interruption;
- server-side archive, retention, hold, disposition, and recovery behavior;
- proof that private signing keys never enter payloads, logs, backups, or fixtures;
- browser-local fallback and export behavior.

Passing this suite establishes client-contract compatibility only. It does not establish production security or legal compliance.
