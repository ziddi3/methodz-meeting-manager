# Methodz Meeting Manager v0.8 Architecture

## Runtime Order

`meeting.html` loads the v0.8 layer after the existing v0.7 modules:

```text
config.js
data-adapter.js
app.js
v0.3-v0.7 feature modules
features-v08-history.js
features-v08-workspace.js
adapter-contract-tests.js
features-v08-accessibility.js
```

The order matters:

- history patches the final saved-record card and save/delete workflows
- workspace backup can see all configured and discovered storage keys
- contract tests use the final adapter API
- accessibility loads last so it can improve dynamic builders and status actions added by earlier layers

## Record Revision Boundary

Revision history is intentionally stored separately from active records.

```text
methodzMeetingRecords
  └─ current active record state

methodzMeetingRevisions
  └─ recordId
      ├─ revision 1 snapshot
      ├─ revision 2 snapshot
      └─ revision n snapshot
```

Reasons for separate storage:

- active record payloads stay readable
- history can be retained or pruned independently
- restoring a revision does not require nested mutation inside the current record
- future remote providers can move revision storage to a dedicated collection

Each revision stores:

```json
{
  "id": "revision-...",
  "recordId": "meeting-...",
  "revisionNumber": 1,
  "capturedAt": "ISO timestamp",
  "reason": "Initial saved record",
  "contentHash": "fnv1a-...",
  "snapshot": {}
}
```

The hash detects package or snapshot identity. It is not a cryptographic signature.

## Archive Lifecycle Boundary

Archive lifecycle is separate from the business-facing meeting `status` field.

```text
Active workspace
  methodzMeetingRecords
        │
        │ Archive
        ▼
Archive Vault
  methodzArchivedMeetingRecords
        │
        ├─ Restore → active workspace
        ├─ Download → JSON
        └─ Permanent Delete → destructive confirmation
```

An archived entry wraps the original record:

```json
{
  "archiveId": "archive-...",
  "archivedAt": "ISO timestamp",
  "originalRecordId": "meeting-...",
  "record": {}
}
```

Revision history remains keyed to the original record ID while the record is archived. Permanent deletion removes both the archived entry and its revision group.

## Workspace Package

Workspace backup operates above the record adapter because it must capture more than meeting records.

The package includes every configured storage key plus discovered browser keys beginning with `methodz`.

```json
{
  "packageType": "methodz-meeting-manager-workspace",
  "packageVersion": 1,
  "appName": "Methodz Meeting Manager",
  "schemaVersion": "0.8.0",
  "exportedAt": "ISO timestamp",
  "entries": {
    "methodzMeetingRecords": "raw JSON string"
  },
  "summary": {},
  "checksum": "fnv1a-..."
}
```

Raw storage strings are preserved instead of reserializing every module's data. This reduces accidental schema changes during backup and restore.

Restore sequence:

1. parse JSON
2. verify package type
3. verify recognized storage entries
4. verify checksum when present
5. show preview
6. request explicit confirmation
7. create a complete local pre-restore recovery package
8. remove current Methodz workspace keys
9. apply package entries
10. reload the app

## Adapter Contract Test Boundary

The contract harness does not run destructive tests against the active adapter.

Instead it:

1. validates the active adapter interface
2. creates a `LocalStorageMeetingAdapter` with a unique temporary key
3. runs CRUD, export, and health checks
4. deletes the temporary key
5. produces a downloadable report

This protects active records while still exercising the real local provider implementation.

The v0.8 required contract remains synchronous:

```text
listRecords()
getRecord(recordId)
replaceRecords(records)
upsertRecord(record)
deleteRecord(recordId)
healthCheck()
```

Optional:

```text
createExportEnvelope(extra)
```

`data-adapter.js` now exposes:

```js
window.MethodzMeetingData.validateAdapter(adapter)
window.MethodzMeetingData.requiredMethods
window.MethodzMeetingData.optionalMethods
```

## Accessibility Layer

The accessibility module is progressive enhancement. Core meeting storage does not depend on it.

Responsibilities:

- add skip navigation
- establish main focus target
- add live status region
- add IDs and label associations to dynamically generated fields
- add keyboard shortcuts
- strengthen focus visibility
- respect reduced-motion preferences

Dynamic builder patching occurs after earlier feature modules have replaced the base functions. The accessibility wrapper therefore preserves the complete function chain.

## Failure Isolation

- Revision read failure returns an empty history map.
- Archive read failure returns an empty archive list.
- Invalid workspace packages are rejected before storage mutation.
- Restore creates a recovery package before deleting current keys.
- Contract tests use isolated temporary keys.
- Accessibility failure does not block record storage.

## Future v1.0 Extraction Targets

The v0.8 layer identifies boundaries that can later become dedicated modules or providers:

```text
RevisionProvider
ArchiveProvider
WorkspacePackageService
SchemaMigrationRegistry
AsyncMeetingDataAdapter
AttachmentStorageAdapter

```

The current implementation stays browser-local and synchronous so it remains compatible with direct-file deployment.
