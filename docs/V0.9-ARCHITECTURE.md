# Version 0.9 Architecture

## Runtime Order

The main workspace loads modules in this order:

```text
config.js
migrations.js
data-adapter.js
app.js
v0.3 through v0.8 feature layers
features-v09-archive.js
features-v09-revisions.js
features-v09-workspace-merge.js
features-v09-pwa.js
```

`migrations.js` is intentionally loaded before `data-adapter.js` and `app.js`. Its `DOMContentLoaded` listener is registered first, so browser-local records are upgraded before the core workspace reads them.

The archive page also loads `migrations.js` before the adapter and renderer.

## Migration Registry

Public API:

```js
window.MethodzMigrations.currentVersion
window.MethodzMigrations.registry
window.MethodzMigrations.compareVersions(left, right)
window.MethodzMigrations.registerMigration(version, description, migrate)
window.MethodzMigrations.migrateRecord(record)
window.MethodzMigrations.migrateWorkspace()
window.MethodzMigrations.validateRecord(record)
window.MethodzMigrations.getState()
```

Migration properties:

- ordered by semantic version
- additive
- idempotent
- unknown-field preserving
- JSON-friendly
- usable without a server or build system

Workspace migration targets:

```text
active records
archived record snapshots
revision snapshots
auto-saved draft
```

Migration state is informational and is stored separately from records.

## Archive Operations

The v0.9 archive layer decorates the existing v0.8 Archive Vault rather than replacing its lifecycle logic.

```text
v0.8 Archive Vault
  ├─ archive
  ├─ restore
  ├─ download
  └─ permanent delete

v0.9 archive operations
  ├─ search
  ├─ filter by status
  ├─ filter by organization
  ├─ select filtered
  ├─ export selected
  └─ export filtered
```

A `MutationObserver` re-applies archive metadata and selection controls whenever the v0.8 vault renderer rebuilds its cards.

## Revision Comparison

The revision comparison module wraps the public v0.8 revision-history opener. It reads snapshots through:

```js
window.getRecordRevisionsV08(recordId)
```

Comparison process:

```text
select left version
select right version
resolve snapshots
flatten nested objects into field paths
compare field identities
classify added / removed / changed
render metrics and table
optionally export JSON
```

The comparison is read-only. It does not create revisions or modify records.

## Workspace Merge

Workspace replacement and workspace merge are deliberately separate operations.

### Replacement restore

```text
v0.8 package restore
  → clear recognized workspace keys
  → write package entries
  → reload
```

### Merge mode

```text
v0.9 package merge
  → validate package and checksum
  → preview conflicts
  → create pre-merge recovery package
  → merge each recognized storage entry
  → preserve local-only keys
  → record merge report
  → reload
```

Merge behavior by storage type:

| Storage type | Merge behavior |
|---|---|
| Active records | ID-aware conflict strategy |
| Archived records | Archive-ID-aware conflict strategy |
| Revision map | Revision identity deduplication and renumbering |
| Directories/templates/presets | Identity-based array merge |
| Settings/metadata objects | Strategy-aware object merge |
| Unknown recognized scalar | Keep local or use incoming according to strategy |

The exported helper object is:

```js
window.MethodzWorkspaceMergeV09
```

It exposes pure merge helpers for browser tests and future extraction.

## PWA Boundary

The PWA layer is optional and protocol-aware.

```text
file: protocol
  → no service worker
  → no install prompt
  → complete direct-file meeting workflow remains available

https: or localhost
  → register service-worker.js
  → cache static app shell
  → expose installation when the browser offers it
```

The service worker never stores meeting records. Record data remains in browser `localStorage`; the worker caches only static application resources.

## Test Architecture

The repository now has two automated validation levels.

### Static validation

- JavaScript syntax checks
- required file checks
- script and stylesheet wiring checks
- manifest JSON parsing

### Browser smoke validation

A temporary Python static server exposes the repository on localhost. Playwright opens the actual workspace and validates core save, migration, archive, merge-helper, manifest, and service-worker behavior.

Playwright is CI-only and is not part of the deployed application.

## New Storage Keys

```text
methodzMigrationState
methodzWorkspaceMergeLog
```

These keys are automatically included in workspace backups because the package builder includes configured keys and discovered keys beginning with `methodz`.

## Design Constraints Preserved

- no runtime framework
- no runtime package manager
- no build command
- direct-file compatibility
- offline-first record ownership
- explicit confirmation before destructive actions
- Methodz described as a brand identity and operating ecosystem
- task responsibility labeled **Assigned To**
