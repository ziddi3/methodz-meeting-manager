# Methodz Meeting Manager v1.6.3 Architecture

## Release boundary

Version 1.6.3 adds a hosted-provider contract and conformance harness while preserving:

- record schema `1.6.0`;
- browser localStorage as the active application provider;
- direct-file operation;
- static hosting;
- no runtime package installation;
- no credential or endpoint configuration.

## Runtime order

The main entry point loads the new modules in this order:

```text
config-v162.js
config-v163.js
migrations-v16.js
provider-contract.js
hosted-provider-adapters.js
data-adapter.js
async-data-adapter.js
attachment-adapter.js
```

`provider-contract.js` has no storage or DOM side effects. `hosted-provider-adapters.js` defines constructors and a factory but does not instantiate a test provider or write records during startup.

## Boundaries

### Existing application storage

The existing synchronous and async adapters remain responsible for the live v1.6.2 application workflow. No existing save, archive, revision, recovery, or export path is silently replaced in v1.6.3.

### Hosted-provider compatibility

The new boundary defines the semantics required before a remote implementation can replace or supplement local storage:

```text
listRecords
getRecord
upsertRecord
archiveRecord
restoreRecord
deleteRecord
exportWorkspace
healthCheck
```

The contract adds behavior that the older compatibility wrapper did not formally define:

- expected conflict tokens;
- idempotency keys;
- Archive Vault operations;
- revision preservation;
- explicit permanent deletion;
- structured retryability metadata;
- private-key-safe provider exports.

### Attachment boundary

Record-provider state contains attachment references already present in meeting records. It does not transfer or cache binary attachments. Binary transport requires a separate provider contract and review.

## Conflict model

Each stored record receives:

```text
providerMetadata.contractVersion
providerMetadata.version
providerMetadata.conflictToken
```

The conflict token is a deterministic canonical-JSON compatibility value. It detects stale client writes but is not a digital signature.

A remote implementation may translate the token to and from a database row version, ETag, transaction sequence, or stronger digest. Observable behavior must remain equivalent:

- create without a prior token;
- update only with the current token;
- reject missing or stale tokens;
- return a new token after an accepted update.

## Idempotency model

The reference providers persist a request fingerprint and accepted result under the caller’s idempotency key.

- identical replay returns the accepted result;
- different input under the same key fails;
- keys are limited to 200 characters;
- production providers must scope keys by tenant and operation.

Idempotency metadata contains no meeting body beyond the existing accepted response in the local reference implementation. A production service should use bounded durable request metadata and must not log sensitive record values.

## Error model

`ProviderError` carries:

```text
code
retryable
operation
providerId
details
```

Retryable errors are limited to temporary categories such as partial failure, unavailability, and rate limiting. Validation, conflicts, authority failures, private-key rejection, and integrity rejection are non-retryable until the caller changes the request or context.

## Export model

`exportWorkspace()` returns a package with:

```text
packageType
packageVersion
providerContractVersion
providerId
exportedAt
activeRecords
archivedRecords
revisions
metadata
integrity
```

The core recursively checks for private EC JWK `d` values and recognized private-key fields before package creation. Integrity is calculated over canonical JSON after all records, revisions, and metadata are assembled.

This compatibility checksum is suitable for deterministic tests and direct-file change detection. Existing cryptographic signing remains a separate optional layer.

## Conformance architecture

`provider-conformance.js` contains one suite that accepts a provider factory. CI invokes the same suite with:

1. `InMemoryHostedProvider`;
2. `LocalStorageHostedProvider` bound to a disposable Storage-compatible object and isolated keys.

The suite never opens the app’s browser localStorage and never reads real records.

The dedicated `provider-conformance` GitHub Actions job is independent from the Playwright matrix. This separates provider behavior failures from browser rendering or engine failures.

## Security boundary

Client conformance does not establish:

- user authentication;
- role or policy authorization;
- tenant isolation;
- encrypted storage;
- network transport security;
- durable delivery;
- append-only audit;
- backup quality;
- data residency;
- server-side retention or legal holds;
- operational incident response.

Those controls belong to any future hosted implementation and must be reviewed before production use.
