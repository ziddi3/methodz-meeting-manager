# Methodz Meeting Manager v1.6 Release Notes

## Release summary

Version 1.6 introduces optional ECDSA P-256 / SHA-256 signatures for exported JSON packages while retaining the static, offline-first architecture and every v1.5 policy and release control.

The release adds a signing workspace inside `meeting.html` and a standalone verifier at `verify.html`.

## Primary additions

- Generate an ephemeral signing key pair in the browser.
- Import a private P-256 JWK into page memory.
- Import and register public P-256 verification keys.
- Download a sensitive private-key backup explicitly.
- Download public verification keys without private material.
- Sign a loaded JSON package and download the signed result.
- Verify signed packages against their embedded public key.
- Detect altered package content and altered signature metadata.
- Mark public keys Active or Revoked in the local registry.
- Export the public-key registry and signing audit.
- Verify a package without importing meeting records through `verify.html`.

## Recommended operating procedure

1. Complete the meeting source record.
2. Apply the correct redaction profile and recipient policy.
3. Obtain a current destination-bound external release approval.
4. Download the approved JSON package and confirm its release receipt.
5. Generate or import the intended signing key.
6. Confirm the signer and key labels.
7. Sign the approved package.
8. Distribute the signed package separately from the private-key backup.
9. Confirm the public key ID through an independent trusted channel.
10. Verify the package with `verify.html` before relying on it.

## Key custody

The private key is never stored by the application. A generated or imported private JWK exists only in the memory of the current page.

Before refreshing or closing the page, download the private backup when continued use of that key is required. Store it in a protected location separate from signed packages, public keys, workspace backups, and routine exports.

Anyone possessing the private backup can produce signatures under the corresponding key ID. Loss of the only private backup prevents future signatures under that key. Exposure requires retiring the key operationally and distributing a replacement public key through a trusted channel.

## Verification meaning

A valid result confirms that the package and displayed signature metadata have not changed since signing and that the signature matches the included public key.

A valid result does not independently establish:

- human identity;
- signer authority;
- approval legitimacy;
- recipient identity;
- delivery;
- legal compliance;
- organization-wide key status.

## Compatibility

- No runtime framework was added.
- No server is required.
- No build step is required.
- Existing records migrate to schema 1.6 through the ordered migration registry.
- Existing v1.5 recipient policies, governance versions, approvals, release receipts, and disposition controls remain active.
- Direct-file meeting workflows remain available.
- Web Crypto signing and verification normally require HTTPS or localhost and depend on browser support.

## Deployment

Deploy the repository as ordinary static files. Ensure these v1.6 assets are published together:

```text
config-v16.js
migrations-v16.js
crypto-package-core.js
features-v16-crypto.js
features-v16-record-metadata.js
features-v16.css
verify.html
verify.js
```

The service worker uses cache name `methodz-meeting-manager-v1.6.0` and includes the complete v1.6 app shell.

## Known trust limitations

Browser-local public-key status and signing audit entries can be changed by someone with access to the browser profile. They are workflow aids, not immutable remote evidence. Durable key governance remains a hosted-provider roadmap item.
