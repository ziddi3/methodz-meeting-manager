# Methodz Meeting Manager v1.6.2 Key Custody Operations

## Release boundary

Version 1.6.2 is an application-shell hardening release. Meeting records remain on schema `1.6.0`.

The release adds a browser-local public-key custody and lifecycle workspace without changing the core static deployment model. It does not create a server, authenticated identity system, hardware security module, certificate authority, or remote immutable ledger.

## Goals

The key custody layer is designed to:

- keep private signing material out of browser storage, backups, manifests, audit exports, and tests;
- record where an approved public verification key is governed;
- record when and how a public-key fingerprint was independently checked;
- provide a documented rotation ceremony that links predecessor and successor keys;
- provide an emergency revocation ceremony with operator, reason, witness, and evidence reference;
- export a public custody manifest with SHA-256 integrity;
- verify and safely merge public custody manifests without downgrading a local revocation;
- preserve direct-file, localhost, and ordinary static-host deployment.

## Browser storage

The release adds two Methodz-prefixed storage collections:

```text
methodzKeyCustodyMetadata
methodzKeyCustodyAudit
```

`methodzKeyCustodyMetadata` is keyed by the derived public key ID. It may contain:

- custodian or controlled role;
- public-key custody-location reference;
- fingerprint verification timestamp;
- fingerprint verifier label;
- independent verification channel;
- next review date;
- non-secret notes;
- metadata update timestamp.

`methodzKeyCustodyAudit` stores a bounded browser-local workflow history. It is not an authenticated or immutable audit ledger.

Neither collection may contain a private JWK, private EC coordinate, password, recovery phrase, token, or other secret.

## Public custody manifest

The exported package type is:

```text
methodz-public-key-custody-manifest
```

A manifest contains:

- package and protocol versions;
- generation timestamp and preparer label;
- operating context;
- public verification keys;
- browser-local lifecycle status;
- predecessor and successor references;
- custody metadata;
- an explicit `privateKeysIncluded: false` declaration;
- a SHA-256 digest over canonical JSON content.

Manifest verification checks:

1. package and protocol version;
2. public-only P-256 JWK structure;
3. derived key ID consistency;
4. duplicate key IDs;
5. lifecycle status and revocation timestamps;
6. custody date formats;
7. prohibited private-key material;
8. SHA-256 package integrity;
9. rotation references included in or absent from the manifest.

A valid manifest proves only that the current public package matches its digest and structural rules. It does not authenticate the named custodian, preparer, operator, witness, organization, authority, or delivery path.

## Fingerprint verification workflow

A fingerprint verification claim requires:

- the selected public key;
- a verifier person or controlled role;
- an independent verification channel;
- a timestamp.

The user must compare the displayed key ID through an independent trusted channel. The application does not perform that independent comparison itself.

Suggested channels include:

- in-person comparison;
- voice-call comparison;
- trusted internal directory;
- signed administrative record;
- another independently verified channel.

## Rotation ceremony

Before rotation:

1. generate or import the successor public key;
2. independently verify its public-key ID;
3. save custody metadata for the successor;
4. select the active predecessor and active successor;
5. record operator, reason, witness, and evidence reference.

Completing rotation:

- marks the predecessor `Revoked`;
- records the revocation timestamp and reason;
- writes `replacedByKeyId` on the predecessor;
- keeps the successor `Active`;
- writes `replacesKeyId` and activation metadata on the successor;
- appends events to both signing and custody audit collections.

The old one-click public-key revoke/restore control is disabled by v1.6.2. Lifecycle changes are routed through the documented custody workflow. Revoked keys are not silently restored.

## Emergency revocation

Emergency revocation requires:

- selected key;
- operator or controlled role;
- documented reason;
- timestamp;
- optional witness;
- optional evidence reference.

The result is browser-local until the updated public custody manifest is exported and distributed through the organization’s trusted channel.

## Verified manifest merge

The merge operation is available only after successful manifest verification.

Merge rules:

- only public JWK material is accepted;
- duplicate IDs are normalized by derived public-key identity;
- missing public keys may be added;
- existing browser-local public keys remain available;
- an existing local `Revoked` status is never downgraded to `Active`;
- incoming `Revoked` status upgrades a matching local entry to revoked;
- private-key material blocks the entire operation;
- the configured key-count limit is enforced;
- a user confirmation is required before mutation.

## Operational checklist

### Scheduled rotation

1. Export the current public custody manifest and signing audit.
2. Generate the successor key pair in a controlled browser session.
3. Download and protect the private backup separately.
4. Independently verify the successor public-key ID.
5. Save successor custody metadata.
6. Complete the rotation ceremony.
7. Export the updated public custody manifest.
8. Verify the exported manifest in a separate browser or device.
9. Distribute the public manifest through the approved internal channel.
10. Preserve the ceremony evidence reference outside browser-local storage.

### Suspected private-key exposure

1. Stop signing with the suspected key.
2. Emergency revoke the public key with a documented reason.
3. Generate a replacement key pair in a controlled session.
4. Independently verify the replacement public key.
5. Complete a rotation from the compromised predecessor to the successor when appropriate.
6. Export and distribute the updated public custody manifest.
7. Review packages signed after the suspected exposure time.
8. Preserve incident evidence and follow organization security procedures.

### Device or browser migration

1. Export a workspace backup.
2. Run a recovery drill.
3. Export the public custody manifest separately.
4. Verify the manifest before importing it on the new device.
5. Do not place the private-key backup in the workspace package.
6. Import private key material into memory only when signing is required.
7. Clear private key material from page memory after the signing session.

## Trust boundary

The v1.6.2 controls are browser-local operational aids. They improve repeatability, safety, and exportable evidence, but they do not replace:

- authenticated user accounts;
- organization-managed key custody;
- hardware-backed private keys;
- a certificate authority;
- server-enforced revocation;
- a remote append-only audit service;
- legal or regulatory review;
- independent security incident response.
