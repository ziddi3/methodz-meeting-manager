# v1.6 Manual Test Checklist

## Runtime

1. Open `meeting.html` on localhost or HTTPS.
2. Confirm the cryptographic package panel is visible and labeled v1.6.
3. Confirm the existing v1.5 policy-operations panel and release-receipt workflows still load.
4. Open the app directly with `file:` and confirm meeting workflows still load even if Web Crypto is unavailable.

## Key generation and backup

1. Enter signer and key labels.
2. Generate a key pair.
3. Confirm a public key appears in the registry.
4. Download private and public key files.
5. Inspect the public file and confirm it contains no `d` private-key field.
6. Clear the private key from memory.
7. Confirm signing is blocked until the private JWK is imported again.

## Public-key registry safety

1. Use the public-key import control to select a private-key backup intentionally.
2. Confirm the stored registry entry contains only public P-256 coordinates.
3. Inspect `methodzSigningPublicKeys` and confirm no registry entry contains `d`.
4. Export the public-key registry and confirm no private material is present.
5. Seed a malformed registry entry, reload, and confirm it is normalized or discarded.

## Signing

1. Create an approved external JSON package through the existing recipient-policy and approval workflow.
2. Load the approved JSON package in the signing panel.
3. Import the private-key backup.
4. Sign and download the package.
5. Confirm `signatureEnvelope` contains the expected key ID, signer label, digest, public key, and signature.
6. Confirm the signed package contains no private key material.
7. Load a private-key backup as the package to sign and confirm signing is blocked.

## Verification

1. Verify the freshly signed package in the main workspace.
2. Confirm signature, digest, and key-ID checks pass.
3. Open `verify.html` and verify the same package.
4. Modify one package field and confirm digest and signature checks fail.
5. Restore the original package, change only `signatureEnvelope.signerLabel`, and confirm the digest still matches but the signature fails.
6. Download a verification report.

## Revocation workflow

1. Mark the public key Revoked.
2. Verify a package signed by that key in the main workspace.
3. Confirm the cryptographic checks may pass but the browser-local registry status makes the overall result invalid.
4. Restore the key and verify again.

## Storage and regression

1. Inspect localStorage and confirm no private JWK is present.
2. Export a workspace backup and confirm it contains no private key material.
3. Save, edit, archive, restore, compare, redact, approve, receipt, and disposition-test a meeting record.
4. Confirm saved records carry schema `1.6.0` and `externalSignatureControl`.
5. Run the complete Playwright suite and the Node Web Crypto self-test.
