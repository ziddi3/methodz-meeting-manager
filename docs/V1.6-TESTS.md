# v1.6 Test Plan

## Automated coverage

### Static validation

GitHub Actions verifies:

- JavaScript and ES module syntax;
- required entry points and v1.6 assets;
- script and stylesheet wiring in `meeting.html`;
- schema wiring in `archive.html`;
- shared core wiring in `verify.html`;
- v1.6 service-worker cache name and assets;
- valid manifest JSON;
- verifier shortcut presence.

### Node Web Crypto self-test

`tests/v16-crypto-core.mjs` runs the portable cryptographic core with Node Web Crypto and confirms:

1. ECDSA P-256 key generation;
2. private and public JWK export;
3. stable key-ID derivation;
4. successful package signing;
5. valid signature verification;
6. payload-tamper rejection;
7. signature-metadata-tamper rejection;
8. absence of private key material from the signed package.

### Playwright coverage

`tests/v16-crypto.spec.js` confirms:

- the v1.6 panel and APIs load;
- schema 1.6 migration registration is active;
- the final record collector is patched;
- a signed package verifies in Chromium;
- changed package content is rejected;
- private material is removed from the local public-key registry;
- sanitation creates an audit event;
- saved records receive `externalSignatureControl` metadata;
- the standalone verifier loads the shared core.

Existing browser suites continue to cover records, archive, revisions, migration, retention, redaction, approvals, disposition, recipient policies, governance, release receipts, and compatibility layers.

## Manual release checklist

### Key generation and backup

- [ ] Open `meeting.html` through HTTPS or localhost.
- [ ] Confirm the v1.6 cryptographic panel reports Web Crypto available.
- [ ] Enter signer and key labels.
- [ ] Generate a key pair.
- [ ] Download the private backup.
- [ ] Confirm the private backup contains `privateKeyJwk` and is clearly marked sensitive.
- [ ] Download the public key.
- [ ] Confirm the public export contains no `d` field.
- [ ] Refresh the page and confirm the private key is no longer loaded.

### Package signing

- [ ] Produce an approved external JSON package through the existing release workflow.
- [ ] Load that package into the signing panel.
- [ ] Import the intended private backup.
- [ ] Sign and download the package.
- [ ] Confirm `signatureEnvelope` includes key ID, labels, timestamp, algorithm, digest, public JWK, trust notice, and signature.
- [ ] Confirm the package contains no private JWK material.
- [ ] Confirm source-record signature metadata updates when a source ID is present.

### Verification

- [ ] Verify the signed package in the main workspace.
- [ ] Verify the same package in `verify.html`.
- [ ] Confirm both report a valid signature.
- [ ] Change one package field and confirm verification fails.
- [ ] Restore the package, change the signer label inside the envelope, and confirm verification fails.
- [ ] Verify with a different public key and confirm key-ID mismatch.
- [ ] Download a verification report and confirm it contains no private material.

### Key registry safety

- [ ] Import a public key and confirm it appears in the registry.
- [ ] Mark the key Revoked and confirm verification reports the local revoked status in the main workspace.
- [ ] Restore the key and confirm the status changes to Active.
- [ ] Export the public registry and confirm every JWK is public-only.
- [ ] Inject or import malformed registry data and confirm the workspace removes unsafe entries.
- [ ] Export the signing audit and confirm generation, import, signing, verification, revocation, and sanitation events are represented.

### Regression and deployment

- [ ] Create, save, edit, archive, restore, revise, and compare a meeting record.
- [ ] Confirm v1.5 recipient-policy review and receipt workflows remain functional.
- [ ] Confirm active preservation holds still block permanent disposition.
- [ ] Confirm direct-file meeting workflows remain usable.
- [ ] Confirm signing and verification work through localhost.
- [ ] Confirm the PWA cache updates to `methodz-meeting-manager-v1.6.0`.
- [ ] Test current Chrome or Chromium, Edge, and Firefox.
- [ ] Test keyboard navigation and screen-reader labels in the signing panel and verifier.

## Failure interpretation

- A digest mismatch indicates package content changed.
- An invalid ECDSA result with a matching digest may indicate altered signature metadata or signature bytes.
- A key-ID mismatch indicates the public key does not match the signed key reference.
- A locally revoked key indicates browser workflow status only. Confirm authoritative key status through the organization’s trusted process.
- Web Crypto unavailable means the browser context cannot perform v1.6 signing or verification. Core meeting workflows should remain usable.
