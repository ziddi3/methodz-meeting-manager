# Methodz Meeting Manager v1.6.1 Recovery Hardening

## Purpose

Version 1.6.1 hardens the existing workspace backup, restore, and merge workflow without changing the meeting-record schema. The release remains a directly deployable static application and preserves direct-file operation.

The new recovery layer addresses three roadmap items:

1. centralized workspace-package validation;
2. safer imports with explicit size and private-key checks;
3. repeatable recovery simulations that do not overwrite browser data.

## Architecture

```text
workspace-package-core.js
  -> package shape and version validation
  -> recognized-key filtering
  -> checksum verification
  -> size limits
  -> private JWK detection
  -> summary consistency checks
  -> restore-plan generation

features-v16-recovery.js
  -> no-write backup inspector
  -> current-workspace dry recovery drill
  -> metadata-only readiness report
  -> browser-local drill history

features-v16-recovery-guards.js
  -> final validation before legacy full restore
  -> final validation before legacy workspace merge
```

The core is environment-neutral. It exposes the same API to the browser and Node-based CI tests. UI code does not duplicate package parsing or restore-plan rules.

## Validation rules

A package is blocked when any of the following is true:

- it is not a JSON object;
- `packageType` is not `methodz-meeting-manager-workspace`;
- `packageVersion` is not `1`;
- `entries` is not a plain object;
- no recognized Methodz storage entries exist;
- a recognized entry is not a string;
- the checksum does not match;
- a configured entry or package size limit is exceeded;
- private JWK material is detected in a parsed storage entry.

The default limits are:

```text
500 storage entries
2 MiB per recognized entry
12 MiB total recognized workspace data
```

Unsupported keys and the recursive pre-restore recovery key are excluded from the restore plan and reported as warnings. Recognized values that are not valid JSON are reported as warnings because older browser storage may contain raw text values.

## Private-key boundary

Workspace backups may contain public-key registry data and signing audit metadata. They must never contain private signing keys.

The validator scans parsed entries for:

- JWK objects containing private fields such as `d`, `p`, `q`, `dp`, `dq`, `qi`, or `oth`;
- explicit `privateKey`, `privateJwk`, or `privateKeyJwk` containers.

A finding blocks restore and merge. This is defense in depth in addition to the v1.6 memory-only private-key rule.

## Recovery readiness panel

The main workspace now includes a **Recovery Readiness** panel after the backup and merge controls.

### Inspect Backup Package

Inspection:

- reads the selected file in memory;
- validates it through the shared core;
- compares incoming entries with current browser storage;
- shows add, replace, unchanged, ignored, and remove counts;
- does not write or delete browser data.

### Run Current Workspace Drill

A drill:

1. creates the same package used by Workspace Backup;
2. validates its checksum, content boundaries, and key safety;
3. simulates a restore into an empty workspace;
4. confirms every recognized entry can be planned for recovery;
5. stores only a compact drill event in `methodzRecoveryDrillLog`.

Drill events do not contain meeting records or workspace values. The log is capped at 100 events.

## Readiness report

The downloadable report contains:

- package metadata;
- validation status, errors, and warnings;
- recognized and ignored storage key names;
- summary counts and byte totals;
- restore-plan key names and counts;
- optional drill event metadata.

It intentionally excludes storage values and meeting content.

## Import guards

The older v0.8 restore and v0.9 merge implementations remain in place for compatibility. Version 1.6.1 wraps their global apply functions and performs shared-core validation immediately before either operation can mutate local storage.

This preserves the existing preview and conflict-resolution interfaces while preventing a package that fails the stronger checks from being applied.

## Recovery drill procedure

For an operational recovery check:

1. Save or update at least one meeting record.
2. Open **Recovery Readiness**.
3. Select **Run Current Workspace Drill**.
4. Confirm the result is **Passed** and checksum verification is reported.
5. Download the readiness report.
6. Export a fresh Workspace Backup.
7. Store the backup and report separately from private signing-key backups.

For a full restore rehearsal, use a separate browser profile or non-production device. Do not overwrite the only working copy of a workspace merely to test recovery.

## Automated coverage

`tests/v16-recovery-core.mjs` covers:

- valid package inspection;
- checksum tampering;
- private JWK rejection;
- entry and package size limits;
- restore-plan classification;
- protected pre-restore-key exclusion.

`tests/v16-recovery.spec.js` covers:

- app-shell and panel wiring;
- legacy restore and merge guards;
- successful current-workspace drills;
- metadata-only drill logs;
- private-key package blocking without local-storage mutation.
