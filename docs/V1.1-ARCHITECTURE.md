# Methodz Meeting Manager v1.1 Architecture

## Purpose

Version 1.1 adds record-lifecycle controls and safe external-copy generation while preserving the v1.0 provider, governance, signature, archive, migration, and recovery boundaries.

## Load Order

The main entry point loads the v1.1 files in this order:

```text
config.js
config-v11.js
migrations.js
migrations-v10.js
migrations-v11.js
provider modules
app.js
historical feature layers
features-v10-governance.js
features-v10-signatures.js
features-v10-release.js
features-v11-retention.js
features-v11-redaction.js
```

The order is intentional:

1. `config-v11.js` extends configuration before the migration registry captures the current schema version.
2. `migrations-v11.js` registers after the v1.0 migration and before `DOMContentLoaded` triggers workspace migration.
3. v1.1 feature layers wrap the final v1.0 data-flow functions.
4. The redaction layer loads after retention so external copies can include a limited retention summary when requested.

The archive entry point uses:

```text
config.js
config-v11.js
migrations.js
migrations-v10.js
migrations-v11.js
data-adapter.js
attachment-adapter.js
archive.js
archive-v10.js
archive-v11.js
```

## Configuration Extension

`config-v11.js` mutates the existing `window.METHODZ_MEETING_CONFIG` object instead of replacing it.

It adds:

- schema and app-shell version `1.1.0`
- retention policy presets
- lifecycle statuses
- redaction profiles
- the redaction-export-log storage key

Keeping the extension separate reduces risk to the stable v1.0 configuration while the feature-layer architecture is still being consolidated.

## Schema Migration

`migrations-v11.js` registers migration `1.1.0` with the ordered registry.

The migration adds:

```text
retentionMetadata
redactionMetadata
schemaAudit.schemaVersion = 1.1.0
releaseMetadata.release = 1.1.0
```

The migration uses object spreads so unknown fields survive. It is safe to run repeatedly because existing values are preferred and defaults fill only absent data.

Validation extends the previous validator rather than replacing it. This keeps v1.0 governance, consent, attachment, and shape checks active.

## Retention Module

`features-v11-retention.js` has four responsibilities.

### Form Integration

The module adds a form panel and wraps:

```text
collectMeetingData()
populateForm()
resetForm()
loadSavedRecords()
createRecordCard()
renderArchiveVaultV08()
```

The wrapper pattern preserves older feature behavior while appending retention metadata.

### Hold Transitions

The module compares the previous hold state with the current checkbox state.

A false-to-true transition creates a `placed` history item.

A true-to-false transition creates a `released` history item.

Repeated saves without a state transition update the current hold details without creating duplicate history events.

### Archive Protection

The Archive Vault remains implemented by the v0.8 history layer. Version 1.1 adds a capture-phase click guard and disables the permanent-delete button when the archived record carries an active hold.

This design adds protection without rewriting the stable archive module.

The control remains client-side. A remote provider must repeat the check server-side.

### Dashboard

The dashboard reads active records from the current adapter-facing helpers and archived entries from the archive storage key.

It does not modify records. Its report export is a metadata index, not a complete workspace backup.

## Redaction Module

`features-v11-redaction.js` treats external sharing as a transformation pipeline:

```text
resolve source
check export permission
clone source
apply profile
strip unsafe keys recursively
append external-copy notice
build redaction manifest
compute integrity digest
preview or download
write export activity metadata
```

### Source Resolution

Sources can be:

- current unsaved form
- active saved record
- archived saved record

Saved sources are cloned before transformation. The controlled source object is never edited.

### Governance Gate

When `MethodzGovernanceV10` is available, external export calls its `canRolePerform(record, "export")` method.

The redaction workflow does not bypass v1.0 policy controls.

### Profile Builders

Each profile creates a new allow-listed object rather than deleting a few fields from a full clone.

After building the object, a recursive unsafe-key pass removes signature, verification, contact, address, file-location, base64, and data-URL fields that may appear in nested or future structures.

This two-stage approach provides both data minimization and defense against schema growth.

### Integrity

The package digest covers the complete unsigned package using stable key ordering.

Preferred algorithm:

```text
SHA-256 through window.crypto.subtle
```

Compatibility fallback:

```text
FNV-1a-32
```

The package labels the exact algorithm. Neither result is presented as signer identity or non-repudiation.

### Export Log

The activity log stores only export metadata. It does not store the redacted package body.

The log is capped at the most recent 200 entries to avoid unbounded localStorage growth.

## Archive Detail

`archive-v11.js` appends retention and preservation information near the record audit.

An active hold receives a visible warning in screen and print output.

## Service Worker

The v1.1 service-worker cache includes new JavaScript, CSS, configuration, migration, and archive assets.

It continues to cache static application resources only. Browser localStorage remains outside the service worker.

## Security Boundaries

The following are workflow controls, not security boundaries:

- local role selection
- legal-hold deletion blocking
- export profile choice
- local activity logs
- local retention review dates

A hosted provider must independently enforce:

- authenticated identity
- authorization
- legal-hold disposition rules
- immutable audit history where required
- secure storage and transport
- server-side redaction policy

## Future Consolidation

The additive layer strategy protects direct-file compatibility but increases wrapper depth.

A future 1.x hardening release can consolidate v1.0 and v1.1 functionality into stable domain services after browser regression coverage proves equivalent behavior.
