# Version 0.7 Architecture

## Runtime Graph

```text
meeting.html
  ├─ config.js
  ├─ data-adapter.js
  ├─ app.js
  ├─ v0.3 workflow modules
  ├─ v0.4 template/archive modules
  ├─ v0.5 attachment/directory modules
  ├─ v0.6 settings/governance modules
  └─ v0.7 organization + navigation modules

archive.html
  ├─ config.js
  ├─ data-adapter.js
  └─ archive.js
```

Both entry points use the same configured record key and active data adapter.

## Data Adapter Boundary

`data-adapter.js` exposes:

```js
window.MethodzMeetingData
```

The adapter manager owns registration and active-provider selection.

The default provider is:

```js
LocalStorageMeetingAdapter
```

Contract:

```js
{
  id,
  label,
  listRecords(),
  getRecord(recordId),
  replaceRecords(records),
  upsertRecord(record),
  deleteRecord(recordId),
  healthCheck()
}
```

Optional provider capability:

```js
createExportEnvelope(extra)
```

The main app's global `getRecords()` and `setRecords()` functions are redirected through the adapter in `features-v07-navigation.js`. This preserves the current feature-module API while separating the storage provider.

## Archive Navigation

Saved record flow:

```text
Saved Record Card
  → session fallback copy
  → archive.html?id=<record-id>
  → adapter.getRecord(record-id)
  → render complete archive
```

The session fallback exists for browser environments where direct local file storage behavior is inconsistent.

Edit handoff:

```text
archive.html
  → sessionStorage: methodzArchiveEditRecordId
  → meeting.html
  → loadRecordForEditing(record-id)
```

Current unsaved preview:

```text
meeting form
  → collectMeetingData()
  → sessionStorage: methodzArchivePreviewRecord
  → archive.html?preview=current
```

Preview mode never writes a meeting record.

## Organization Directory

Storage key:

```text
methodzOrganizationDirectory
```

Directory entry:

```json
{
  "id": "organization-...",
  "name": "Example Partner",
  "type": "Partner Organization",
  "primaryRepresentative": "Name",
  "contact": "Contact detail",
  "notes": "Context",
  "createdAt": "ISO timestamp",
  "updatedAt": "ISO timestamp"
}
```

Meeting snapshot:

```json
{
  "organizationDetails": [
    {
      "id": "organization-...",
      "name": "Example Partner",
      "type": "Partner Organization",
      "primaryRepresentative": "Name",
      "contact": "Contact detail",
      "notes": "Context"
    }
  ]
}
```

The snapshot protects historical meaning. Changing or deleting a directory entry does not rewrite old meetings.

## Print Model

The archive page is the preferred full-record print surface.

`features-v07.css`:

- hides navigation and management controls
- keeps record sections visible
- preserves attendance, signature audit, attachment, and task tables
- discourages page breaks inside logical record cards
- reduces audit typography for dense print output

## Cloud Provider Path

A future provider should implement the adapter contract and register before the meeting workflow starts.

Example shape:

```js
const cloudAdapter = {
  id: "methodz-api",
  label: "Methodz API",
  listRecords() {},
  getRecord(recordId) {},
  replaceRecords(records) {},
  upsertRecord(record) {},
  deleteRecord(recordId) {},
  healthCheck() {}
};

window.MethodzMeetingData.registerAdapter(cloudAdapter);
window.MethodzMeetingData.useAdapter("methodz-api");
```

For asynchronous cloud APIs, the v1.0 adapter contract will need an asynchronous compatibility layer. Version 0.7 intentionally keeps the current synchronous offline core stable.

## Compatibility Rules

- Preserve unknown imported fields.
- Treat all new record sections as optional when reading.
- Never store reusable signatures in directories.
- Keep Attachment References as metadata until an attachment provider exists.
- Keep Methodz described as a brand identity and operating ecosystem.
- Keep responsibility labels as **Assigned To**.
