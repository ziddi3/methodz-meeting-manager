# Methodz Meeting Manager v1.4 Release Notes

## Release theme

Version 1.4 adds recipient-specific external-export policies and minimum field allow-lists without adding a server, runtime dependency, or build step.

The existing redaction profile remains the first safety boundary. A recipient policy is applied afterward and may remove more information, but it cannot restore information already removed by Partner Safe, Public Summary, or Custom External Copy.

## Added

- Browser-local Recipient-Specific Export Policy registry.
- Named recipient or accountable-contact field.
- Organization and contact-reference fields.
- Base destination-policy selection.
- Allowed redaction-profile selection.
- Maximum external field groups per recipient.
- Policy status and review date.
- Sensitive discussion-note safeguard requiring a meaningful verification note.
- Unique dynamic destination IDs in the form `recipient:<policy-id>`.
- Recipient-policy snapshots in redaction manifests and approval records.
- Policy JSON import/export.
- Recipient-policy audit JSON export.
- Schema 1.4 record metadata and migration.
- Playwright coverage for policy creation, field filtering, approval binding, sensitive-note validation, and overdue-policy blocking.

## Field allow-list groups

- Core meeting information
- Organizations / representatives
- Attendee names and roles
- Agenda
- Discussion notes, explicitly marked sensitive
- Decisions
- Follow-Up Tasks
- Attachment metadata
- Meeting summary
- Retention summary

Core meeting information is always included in the recipient policy. The selected redaction profile may still remove or transform fields within any allowed group.

## Approval binding

When a recipient policy is applied, v1.4 creates a unique destination-policy ID from the policy ID. The existing v1.2 approval fingerprint already includes the destination-policy ID and redacted content. This means an approval for one named recipient policy does not authorize a different recipient policy, even when the visible content is otherwise identical.

The approval record also stores a recipient-policy snapshot for audit and review.

## Review-date behavior

An active policy that is past its review date cannot be applied, previewed, or used to request approval. Update the policy and record a new review before continuing.

A missing review date is allowed for compatibility, but an explicit review date is recommended for every external recipient.

## Security boundary

Recipient policies, requester names, reviewer names, approvals, and audit events are browser-local workflow controls. They are not authenticated identities, proof of delivery, immutable audit records, or legal signatures.

A future hosted provider must enforce recipient authorization, field allow-lists, approval state, and audit retention on the server.

## Deployment

Version 1.4 remains deployable by opening `meeting.html` directly or by publishing the repository to an ordinary static host. Hosted HTTPS or localhost deployments may use the optional service worker.
