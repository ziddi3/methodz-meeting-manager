# v1.6.4 Hosted-Provider Pilot Harness

## Purpose

Version 1.6.4 exercises the v1.6.3 hosted-provider contract across a serialized, HTTP-style request boundary while connecting **no backend**, production endpoint, credential, tenant directory, or remote database.

The harness is CI-only evidence. It is not loaded by `meeting.html`, `archive.html`, `verify.html`, or the service worker. The application therefore remains a plain static, offline-first HTML/CSS/JavaScript deployment.

## Components

### `http-provider-pilot.js`

Exports:

- `HttpProviderSimulator`
- `HttpHostedProviderClient`
- `createPilotProvider()`
- diagnostic redaction helpers

The simulator owns disposable per-tenant `InMemoryHostedProvider` instances. The client converts provider calls into JSON-safe request envelopes and converts the simulator response back into provider results or `ProviderError` objects.

The boundary deliberately performs JSON serialization in both directions. This catches assumptions that only work when a client and provider share JavaScript object identity.

### `tests/v164-provider-pilot.mjs`

Runs the complete v1.6.3 provider conformance suite against the HTTP-style client, then adds transport-specific scenarios for:

- rate limiting and retry;
- response loss after a successful commit;
- partial-success replay;
- request timeout and retry;
- tenant-scoped idempotency;
- serialized preservation of revisions and governance metadata;
- diagnostics that exclude record content, credentials, tokens, and private key material.

### `.github/workflows/provider-pilot.yml`

Runs the pilot independently from browser regression checks. It installs no npm packages and adds no deployed dependency.

## Request envelope

A request crossing the pilot boundary contains:

```json
{
  "protocol": "methodz-provider-pilot/1.0",
  "requestId": "pilot:opaque-id",
  "tenantId": "synthetic-tenant",
  "operation": "upsertRecord",
  "attempt": 0,
  "body": {}
}
```

The request body is never included in simulator diagnostics.

## Response envelope

Successful response:

```json
{
  "ok": true,
  "requestId": "pilot:opaque-id",
  "result": {}
}
```

Failed response:

```json
{
  "ok": false,
  "requestId": "pilot:opaque-id",
  "error": {
    "name": "ProviderError",
    "code": "UNAVAILABLE",
    "retryable": true,
    "operation": "upsertRecord",
    "details": {}
  }
}
```

Error details are sanitized before crossing the boundary. Keys associated with credentials, private material, signatures, record content, or conflict tokens are redacted.

## Retry and idempotency behavior

The client retries only errors marked `retryable`. Retry count and timeout are explicit client settings.

The pilot demonstrates the most dangerous hosted-write case:

1. the provider commits an upsert;
2. the response is lost;
3. the client receives `UNAVAILABLE` or `PARTIAL_FAILURE`;
4. the client retries with the same idempotency key;
5. the tenant provider replays the original result instead of creating a second write.

Idempotency storage is tenant-scoped because every synthetic tenant has an isolated provider state. The same idempotency key can be used independently by two tenants without collision.

## Fault model

The simulator supports queued, one-shot faults:

- `rateLimit`
- `unavailable`
- `delay`
- `dropResponse`
- `partialSuccess`
- a supplied `ProviderError`

Faults may occur before dispatch or after a successful provider operation. This makes uncertain-write and retry behavior testable without real network infrastructure.

## Metadata preservation

The serialized pilot verifies preservation of:

- active and archived record separation;
- revisions;
- attachment references;
- unknown future fields;
- retention metadata;
- preservation holds;
- disposition metadata;
- redaction and approval metadata;
- release receipts;
- signature metadata;
- public-key custody metadata;
- recovery-package and source-integrity metadata.

Binary attachment bytes, private JWK parameters, provider credentials, access tokens, and embedded `data:` payloads remain rejected by the v1.6.3 contract.

## Diagnostic boundary

Pilot diagnostics contain only:

- opaque request ID;
- operation name;
- hashed tenant reference;
- success or failure status;
- error code;
- duration;
- timestamp.

They do not contain meeting titles, notes, summaries, record IDs, signatures, authorization data, access tokens, private keys, JWK material, request bodies, or response bodies.

The pilot log is process-local and disposable. It is not a durable audit ledger.

## Deployment boundary

The pilot does not:

- select a production provider;
- contact Firebase, Supabase, Google Drive, a CRM, or a Methodz API;
- implement authentication or authorization;
- store secrets;
- alter the record schema;
- change the default localStorage data adapter;
- weaken retention, preservation, redaction, approval, receipt, signature, custody, disposition, or recovery safeguards.

A real hosted provider must pass the same conformance and transport scenarios and provide the operational evidence in `PRODUCTION-PROVIDER-EVIDENCE.md` before it can be considered for production use.
