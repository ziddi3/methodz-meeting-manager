# Methodz Meeting Manager v1.3 Release Notes

## Release theme

Version 1.3 adds controlled permanent-disposition review and a browser-local preservation event chain while keeping the application static, offline-first, dependency-free at runtime, and directly deployable from `meeting.html`.

## Permanent disposition approval

Archive Vault permanent removal now requires all of the following:

1. the archived record has no active preservation hold;
2. a requester documents the retention review and disposition basis;
3. a different authorized reviewer approves the request;
4. the approval fingerprint still matches the archived record;
5. the existing final confirmation is accepted.

Default approval roles are Administrator and Auditor. Browser-local role context is a workflow control, not authenticated identity.

Disposition requests can be pending, approved, rejected, revoked, or consumed. A successful removal consumes the approval so it cannot be silently reused.

## Separation of duties

The default workflow requires the reviewer name to differ from the requester name. This is a local workflow safeguard only. A future hosted provider must enforce authenticated role and separation-of-duty rules on the server.

## Preservation event chain

The app records these event types when they occur:

- legal hold placed;
- legal hold released;
- disposition requested;
- disposition approved;
- disposition rejected;
- disposition revoked;
- approved archive removal completed.

Each event includes the prior digest and its own FNV-1a-32 local chain checksum. The app can verify the complete sequence and export it as JSON.

This chain is tamper-evident only inside the current browser storage. It is not immutable, authenticated, cryptographically signed, or equivalent to a server audit ledger.

## Fingerprint safety

Disposition approval is bound to a deterministic archived-record fingerprint. Volatile schema, adapter, synchronization, and release-audit metadata is excluded. A record change invalidates the approval and requires a new request.

The fingerprint is an identity checksum, not a digital signature.

## Archive detail

The dedicated archive page now shows disposition approvals and preservation events associated with the record, plus the current local chain verification result.

## Compatibility

- Existing v1.2 export approvals remain intact.
- Existing v1.1 retention and legal-hold rules remain stronger than disposition approval.
- Existing records migrate forward to schema 1.3.0.
- Direct-file operation remains supported.
- No framework, build step, cloud endpoint, or runtime package was added.
