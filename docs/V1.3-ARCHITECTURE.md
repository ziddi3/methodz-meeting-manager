# Methodz Meeting Manager v1.3 Architecture

## Load order

Version 1.3 extends the existing layered static architecture:

```text
config.js
config-v11.js
config-v12.js
config-v13.js
migrations.js
migrations-v10.js
migrations-v11.js
migrations-v12.js
migrations-v13.js
...
features-v11-retention.js
features-v12-export-approval.js
features-v12-fingerprint-policy.js
features-v12-release-audit.js
features-v12-compatibility.js
features-v13-disposition.js
```

The v1.3 module loads last so it can preserve the v1.1 legal-hold gate and add a second gate around the existing Archive Vault lifecycle.

## Configuration

`config-v13.js` sets schema and app-shell version `1.3.0` and adds:

```text
methodzDispositionApprovals
methodzDispositionAuditLog
methodzPreservationEventChain
```

It declares allowed review roles, separation-of-duty behavior, and local registry limits.

## Schema

`migrations-v13.js` adds:

```json
{
  "dispositionControl": {
    "approvalRequired": true,
    "lastRequestId": "",
    "lastApprovedAt": "",
    "lastDispositionAt": "",
    "preservationEventHead": ""
  }
}
```

Unknown record fields remain preserved by the ordered migration registry.

Approval requests and event-chain entries remain in separate browser-local registries so review actions do not rewrite the controlled meeting content.

## Archive gate order

The effective order is:

1. v1.1 preservation-hold validation;
2. v1.3 disposition approval and fingerprint validation;
3. existing Archive Vault final confirmation;
4. existing archive and revision removal;
5. v1.3 approval consumption and completion event.

An active preservation hold always wins over a disposition approval.

## Disposition fingerprint

The fingerprint uses stable key ordering and an FNV-1a-32 checksum. Volatile metadata is removed before calculation.

Purpose:

- detect whether the archived record changed after review;
- prevent an approval for one archive entry from authorizing another;
- bind approval to the reviewed local record state.

It is not a digital signature or identity proof.

## Event chain

Every event contains:

```text
sequence
action
occurredAt
sourceRecordId
archiveId
meetingNumber
title
actor
roleContext
note
details
previousDigest
algorithm
digest
```

The verifier recomputes every event digest and checks every previous-digest link.

Because localStorage is editable, the chain is only a local tamper-evidence aid. A hosted provider must store events append-only and enforce authenticated authorization on the server.

## Archive detail extension

`archive-v13.js` reads disposition approvals and preservation events associated with the displayed record ID. It does not alter the record.

## Remote-provider requirements

A future Firebase, Supabase, CRM, or Methodz API provider must independently enforce:

- authenticated requester and reviewer identities;
- authorization by role;
- separation of duties;
- active preservation holds;
- record version or fingerprint matching;
- append-only audit storage;
- durable approval consumption;
- server-side destructive action controls.
