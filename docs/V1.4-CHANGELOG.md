# Methodz Meeting Manager v1.4 Changelog

## Added

- `config-v14.js` with schema 1.4, recipient-policy storage keys, and a configurable field-group catalog.
- `migrations-v14.js` with `externalRecipientControl` defaults and validation.
- `features-v14-recipient-policy.js` with a browser-local named-recipient policy registry.
- Dynamic destination-policy IDs in the form `recipient:<policy-id>`.
- Per-recipient allowed redaction profiles.
- Per-recipient maximum external field allow-lists.
- Policy status, review date, verification note, contact reference, import/export, and audit export.
- Sensitive discussion-note validation.
- Recipient-policy snapshots in redaction manifests and approval records.
- `features-v14.css`.
- `tests/v14-smoke.spec.js`.
- v1.4 release, architecture, and manual-test documentation.

## Changed

- Bumped the active schema and app-shell version to `1.4.0`.
- Loaded v1.4 configuration and migration assets in both `meeting.html` and `archive.html`.
- Loaded the v1.4 feature and stylesheet in the main workspace.
- Updated the optional service-worker cache to `methodz-meeting-manager-v1.4.0`.
- Extended GitHub Actions static checks and browser smoke coverage for v1.4.
- Refined external-export governance from destination-level section rules to named-recipient field allow-lists.

## Security behavior

- Recipient policies are subtractive and run after redaction.
- A recipient policy cannot restore a field removed by the selected redaction profile.
- Approval fingerprints remain bound to redacted content and now use a recipient-specific destination-policy ID.
- Overdue or inactive recipient policies cannot be used.
- Typed signatures and signature-verification details remain excluded from all external copies.
- Browser-local policies, names, approvals, and audit events are not authenticated identities or immutable compliance records.

## Compatibility

- Direct-file use remains supported.
- No runtime dependency, package manager, cloud endpoint, or build step was added.
- Generic v1.2 destination policies remain available for compatibility.
