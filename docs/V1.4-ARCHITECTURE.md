# Methodz Meeting Manager v1.4 Architecture

## Purpose

Version 1.4 introduces a recipient-policy layer between redaction and approval.

```text
Controlled source record
  -> v1.1 redaction profile
  -> v1.4 recipient field allow-list
  -> package integrity calculation
  -> v1.2 fingerprint-bound approval
  -> approved external download
```

The order is deliberate. Recipient policies are subtractive only.

## Modules

```text
config-v14.js
  Schema/app-shell 1.4 configuration
  Recipient-policy storage keys
  Field-group catalog

migrations-v14.js
  externalRecipientControl defaults
  Schema validation extension

features-v14-recipient-policy.js
  Policy registry and editor
  Dynamic destination-policy generation
  Field allow-list enforcement
  Approval-request policy snapshots
  Import/export and audit events

features-v14.css
  Policy editor, status, card, mobile, focus, and print presentation
```

## Script-order contract

`config-v14.js` must load after `config-v13.js` and before `migrations.js`.

`migrations-v14.js` must load after the base migration registry and v1.3 migration.

`features-v14-recipient-policy.js` must load after the v1.1 redaction modules, v1.2 approval modules, and v1.3 disposition module. Its DOM-ready callback wraps the final external-preview and approval-request functions exposed by the earlier layers.

## Dynamic destination policies

Every active recipient policy becomes a runtime destination policy:

```text
recipient:<policy-id>
```

The runtime entry carries the policy's permitted profiles and permitted Custom External Copy sections. Existing v1.2 validation therefore rejects a disallowed profile or custom section before approval.

Because the v1.2 content fingerprint includes `destinationPolicyId`, approvals are recipient-policy-specific.

## Field filtering

The field catalog maps policy groups to redacted package top-level fields. After the v1.1 redaction result is prepared, v1.4 removes every top-level field not listed by the selected recipient policy.

The operation mutates only the temporary external package. It never changes the controlled source record.

After filtering, package integrity is recalculated. The redaction manifest gains:

```text
manifest.recipientPolicy
manifest.removedPaths
manifest.warnings
```

## Sensitive discussion notes

A policy cannot enable the `discussion-notes` field group unless `verificationNote` contains a meaningful explanation. This is a workflow safeguard against casually enabling the broadest free-form content field.

The selected base destination and redaction profile may still forbid discussion notes.

## Migration metadata

Schema 1.4 adds:

```text
externalRecipientControl.policyRequired
externalRecipientControl.defaultRecipientPolicyId
externalRecipientControl.lastRecipientPolicyId
externalRecipientControl.lastRecipientPolicyLabel
externalRecipientControl.lastRecipientPolicyAppliedAt
```

The migration preserves unknown fields and all earlier governance metadata.

## Trust boundary

The local registry improves consistency and reviewability, but it is not an authorization service. Anyone with browser-storage access may change policies or logs.

A remote provider should eventually supply:

- authenticated recipient and reviewer identities;
- server-enforced field allow-lists;
- immutable or append-only audit storage;
- server-enforced approval expiry and revocation;
- delivery logging and recipient verification;
- organization-managed policy administration.
