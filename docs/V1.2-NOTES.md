# Methodz Meeting Manager v1.2 Notes

## Release focus

Version 1.2 adds controlled external-release approval to the existing partner-safe export workflow.

The controlled meeting record remains unchanged. Approval requests, reviewer decisions, download history, and destination policies are stored separately in browser-local workflow data.

## New capabilities

- Approval request for the exact redacted content fingerprint.
- Intended-destination policy selection.
- Named requester and reviewer sign-off.
- Approval expiry dates.
- Approve, reject, and revoke transitions.
- Approved JSON and HTML packages containing release metadata.
- Package integrity recalculated after approval metadata is attached.
- Browser-local approval and release audit export.
- Saved-record metadata showing the last approved external release when the source is an active record.

## Default destination policies

### Canadian Soft Water Corporation

Allows Partner Safe or Custom External Copy. Custom copies may include attendee names and roles, agenda, decisions, tasks, attachment metadata, and limited retention details. Discussion notes remain blocked by default.

### Method HVAC Inc.

Uses the same conservative internal-partner allow-list as Canadian Soft Water Corporation.

### Public / Website

Allows only the Public Summary profile.

### Other External Recipient

Allows Partner Safe or Public Summary. Custom exports are blocked by default.

These are configurable workflow presets, not legal conclusions about what a recipient is entitled to receive.

## Approval identity boundary

The app records typed requester and reviewer names, timestamps, notes, expiry, and content fingerprints. It does not authenticate identity, create an immutable audit ledger, or provide a legally qualified electronic signature.

A future hosted provider must enforce authentication, authorization, destination policy, approval separation, legal holds, and retention requirements on the server.

## Direct-file compatibility

The feature remains compatible with opening `meeting.html` directly. Web Crypto SHA-256 is preferred where available. The existing clearly labeled compatibility checksum remains available when Web Crypto cannot run.
