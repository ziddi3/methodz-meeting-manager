# v1.0 Architecture

## Runtime Order

The main workspace loads modules in this order:

1. `config.js`
2. `migrations.js`
3. `migrations-v10.js`
4. `data-adapter.js`
5. `async-data-adapter.js`
6. `attachment-adapter.js`
7. `app.js`
8. historical feature layers
9. `features-v10-governance.js`
10. `features-v10-signatures.js`
11. `features-v10-release.js`

The ordering is intentional.

The v1.0 migration extension must register before `DOMContentLoaded`, and the v1.0 feature modules must patch the final versions of older data-flow functions.

## Governance Boundary

`features-v10-governance.js` evaluates a local role against:

1. the selected policy action matrix
2. the record-level `allowedRoles` list

This boundary can block UI operations, but it cannot authenticate a person. A remote adapter must repeat authorization checks server-side.

## Synchronous and Asynchronous Providers

The synchronous provider remains the immediate record source used by the current interface.

The asynchronous contract is separate so future providers can support:

- network latency
- remote failures
- retries
- conflict tokens
- authenticated requests
- eventual consistency

The default async adapter simply wraps the local synchronous adapter.

## Attachment Boundary

Meeting records store attachment metadata, not file bytes.

A future binary provider should return stable references and preserve these fields:

- reference ID
- display label
- type
- external or provider location
- date
- added by
- notes

Binary payloads must not be serialized into the meeting record.

## Consolidated Workspace

The v1.0 records workspace reads:

- active records through the current record adapter
- archived entries from the archive storage key

It does not merge the active and archived storage collections. The combined display is a read-oriented index.

## Release Audit

Release validation is intentionally stricter than schema migration.

Migration answers: “Can this record be normalized safely?”

Release validation answers: “Is this record complete enough for the current workflow?”

Warnings do not prevent storage. Signature consent errors do prevent saving a typed signature.
