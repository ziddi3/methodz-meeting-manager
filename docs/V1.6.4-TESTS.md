# v1.6.4 Hosted-Provider Pilot Test Plan

## Automated release gate

Run:

```bash
node tests/v164-provider-pilot.mjs
```

Expected result:

- process exits with code 0;
- JSON output reports `ok: true`;
- the complete reusable provider conformance suite passes;
- all transport-specific scenarios are listed.

GitHub Actions runs the same command in `.github/workflows/provider-pilot.yml` with Node.js 22.

## Covered scenarios

### Contract compatibility

- provider shape and contract version;
- health reporting;
- empty list and missing-record behavior;
- idempotent create;
- expected-conflict-token updates;
- revisions;
- archive and restore;
- permanent-delete guard;
- provider export integrity;
- unsafe write and export rejection.

### Transport behavior

- request and response JSON round-trip;
- retry after `RATE_LIMITED`;
- response loss after committed write;
- `PARTIAL_FAILURE` after committed write;
- idempotent replay after uncertain outcome;
- timeout before a read response;
- bounded client retry count;
- retryable versus non-retryable error handling.

### Tenant behavior

- separate provider state per tenant;
- same record ID may exist independently in two tenants;
- same idempotency key may be used independently in two tenants;
- tenant references in diagnostics are hashed.

### Metadata preservation

- attachment references;
- unknown future fields;
- revisions;
- retention classification;
- preservation holds;
- disposition state;
- redaction and approval metadata;
- release receipts;
- signature metadata;
- public-key custody metadata;
- recovery and integrity metadata.

### Diagnostic safety

- meeting title and notes do not enter diagnostics;
- record IDs do not enter diagnostics;
- private JWK values do not enter diagnostics;
- request and response bodies do not enter diagnostics;
- errors preserve only sanitized operational metadata.

## Static deployment regression

The workflow verifies that `http-provider-pilot.js` is not referenced by:

- `meeting.html`;
- `archive.html`;
- `verify.html`;
- `service-worker.js`.

This keeps the simulator out of the deployed runtime and app-shell cache.

## Manual review

Before merging:

1. Confirm `config-v164.js` is loaded after `config-v163.js` in `meeting.html` and `archive.html`.
2. Confirm the service-worker cache is `methodz-meeting-manager-v1.6.4`.
3. Confirm `config-v164.js` is cached.
4. Confirm the pilot module itself is not cached or loaded by an application page.
5. Confirm the record schema remains `1.6.0`.
6. Review `PRODUCTION-PROVIDER-EVIDENCE.md` for completeness.
7. Confirm no endpoint, token, API key, client secret, or production project identifier was committed.

## Existing regression suites

The existing workflows must continue to pass:

- Static and Browser Checks
- Hosted Provider Conformance
- portable signature verification
- recovery validation
- custody validation

The pilot is additive. It must not replace existing direct-provider, browser, cryptographic, recovery, or custody tests.
