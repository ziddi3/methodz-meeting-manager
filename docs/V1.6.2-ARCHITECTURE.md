# v1.6.2 Public-Key Custody Architecture

## Layering

The v1.6.2 implementation is additive:

1. `crypto-package-core.js` remains the authority for P-256 public JWK normalization and key-ID derivation.
2. `key-custody-core.js` builds and validates portable public custody manifests.
3. `features-v162-custody.js` provides the browser UI, bounded local event storage, and downloads.
4. `config-v162.js` changes only app-shell and custody configuration. It deliberately does not change the meeting-record schema.

## Storage

The browser-local event collection uses:

```text
methodzSigningCustodyEvents
```

The collection is bounded by `METHODZ_MEETING_CONFIG.keyCustody.maximumEvents`. Each entry contains only:

- event ID and type
- planned, completed, or cancelled state
- source and replacement public-key IDs
- effective and recorded timestamps
- operator and witness labels
- reason or evidence note
- checklist booleans

No private JWK is accepted or written.

## Public custody manifest

The portable package type is:

```text
methodz-public-key-custody-manifest
```

It includes normalized public JWKs, derived key IDs, key lifecycle metadata, and normalized custody events. Manifest validation recalculates each public key ID and validates every event reference.

## Fail-closed rules

Validation fails when:

- a public key contains a private `d` parameter
- a key ID does not match the included public JWK
- an event references an unknown source or replacement key
- a rotation reuses the same key as source and replacement
- an event has an invalid effective date
- operator or evidence text is missing
- a completed event lacks a witness
- any completed-event checklist item is false

## Registry separation

Custody records do not mutate the v1.6 public-key registry. This prevents a documentation action from silently changing verification behavior. Operators must explicitly revoke or restore a key using the existing registry control, then record the corresponding custody event.

## Hosted provider boundary

A future hosted provider may synchronize public keys and custody events only if it preserves these invariants:

- key IDs are always derived from normalized P-256 public JWKs
- private JWKs never enter provider payloads, logs, backups, or telemetry
- provider status is not treated as identity proof
- offline exports remain valid independently of the provider
- direct-file meeting workflows remain available
