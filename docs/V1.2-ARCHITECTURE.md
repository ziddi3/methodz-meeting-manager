# Methodz Meeting Manager v1.2 Architecture

## Load order

```text
config.js
config-v11.js
config-v12.js
migrations.js
migrations-v10.js
migrations-v11.js
migrations-v12.js
...
features-v11-redaction.js
features-v11-redaction-policy.js
features-v12-export-approval.js
```

The v1.2 layer loads after the v1.1 redaction policy so it can gate both existing external-download buttons without changing the stable v1.1 implementation.

## Storage

```text
methodzExternalExportApprovals
methodzExternalExportApprovalLog
```

`methodzExternalExportApprovals` stores request state and release history. `methodzExternalExportApprovalLog` stores append-style browser-local workflow events.

Neither key is an immutable compliance ledger.

## Content fingerprint

Approval binds to a stable fingerprint over:

```json
{
  "fingerprintType": "methodz-external-content-v1",
  "destinationPolicyId": "...",
  "profile": "...",
  "record": {}
}
```

The preview generation timestamp and package export timestamp are intentionally excluded. This lets the approved content be downloaded later while still invalidating approval whenever the redacted meeting content, destination, or profile changes.

## Approved package

The approved package adds:

```text
packageType: methodz-approved-external-meeting-copy
packageVersion: 2
approval.approvalId
approval.destinationPolicyId
approval.requestedBy
approval.approvedBy
approval.approvedAt
approval.expiresAt
approval.purpose
approval.contentFingerprint
```

Package integrity is recalculated after approval metadata is attached.

## Policy validation

Destination policy validation occurs during:

1. approval request creation
2. reviewer approval
3. approved package download

Custom sections are checked against the destination allow-list. Signature data remains excluded by the v1.1 redaction layer regardless of destination.

## Record integration

The v1.2 migration adds `externalReleaseControl` to meeting records:

```text
approvalRequired
defaultDestinationPolicyId
lastApprovalId
lastApprovedAt
lastExportAt
```

Approval objects remain separate from controlled meeting content. When an approved active record is downloaded, only the last-release metadata is copied into the active source record.

## Future provider requirements

A remote provider must independently enforce:

- authenticated reviewer identity
- role and permission checks
- separation of requester and approver where required
- approval expiry
- destination allow-lists
- legal-hold restrictions
- immutable or tamper-evident audit storage
- server-side package signing when key management exists
