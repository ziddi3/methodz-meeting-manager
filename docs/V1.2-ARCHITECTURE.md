# Methodz Meeting Manager v1.2 Architecture

## Load order

The v1.2 extension loads after the v1.1 retention and redaction modules:

```text
config.js
config-v11.js
config-v12.js
migrations.js
migrations-v10.js
migrations-v11.js
migrations-v12.js
...
features-v11-retention.js
features-v11-redaction.js
features-v11-redaction-policy.js
features-v12-approvals.js
```

This order matters. The v1.2 module wraps the v1.1 external-export functions and recomputes integrity after approval metadata is added.

## Configuration extension

`config-v12.js` sets schema and app-shell version `1.2.0` and adds:

```text
methodzExternalExportApprovals
methodzExternalExportApprovalLog
methodzDispositionApprovals
methodzDispositionLog
```

It also declares approval roles and recipient-policy profile allow-lists.

## Schema migration

`migrations-v12.js` adds `approvalMetadata` to records and preserves all unknown fields.

```json
{
  "approvalMetadata": {
    "exportApprovalRequired": true,
    "dispositionApprovalRequired": true,
    "lastApprovedExportId": "",
    "lastApprovedExportAt": "",
    "lastDispositionApprovalId": "",
    "lastDispositionApprovedAt": "",
    "updatedAt": "ISO timestamp"
  }
}
```

The approval registries themselves stay outside controlled meeting records so that approval changes do not rewrite source meeting content.

## Fingerprints

Approval fingerprints use a deterministic stable JSON representation and FNV-1a identity checksum. Volatile fields such as timestamps, release metadata, schema audit, and adapter metadata are removed before fingerprinting.

The fingerprint is used to detect source changes. It is not a cryptographic signature.

External package integrity remains SHA-256 through Web Crypto where available, with the existing labeled compatibility fallback in direct-file environments.

## External-export wrapper

The v1.2 module wraps:

```text
previewExternalExportV11()
downloadExternalJsonV11()
downloadExternalHtmlV11()
```

Preview remains available before approval. Download is blocked until approval is valid for the current fingerprint, recipient policy, and profile.

After approval metadata is attached to a preview package, package integrity is recomputed so the digest covers the approval snapshot.

## Disposition gate

The v1.2 module observes Archive Vault delete actions in capture phase.

Evaluation order:

1. v1.1 preservation-hold protection;
2. v1.2 disposition approval validation;
3. existing Archive Vault confirmation and deletion;
4. v1.2 completion audit and approval consumption.

## Archive detail

`archive-v12.js` adds a read-only Approval Governance section showing browser-local export and disposition approval records associated with the meeting record ID.

## Remote-provider requirements

A remote provider must not trust browser-local role context or approval registries. It must independently validate permissions, source versions, active holds, approval status, and audit durability on the server.
