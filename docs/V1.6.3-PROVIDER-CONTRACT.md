# Methodz Meeting Manager v1.6.3 Hosted-Provider Contract

## Purpose

Version 1.6.3 defines the client-side contract that a future Firebase, Supabase, Drive, CRM, or Methodz API provider must satisfy. It does **not** add a backend, authentication system, remote endpoint, credential, or runtime dependency.

The existing browser-local workflows remain the default. The record schema remains **1.6.0** and the application remains deployable as plain static files.

## Portable modules

```text
provider-contract.js          Contract constants, errors, integrity helpers
hosted-provider-adapters.js   Disposable memory and localStorage reference providers
provider-conformance.js       Reusable provider test suite
```

All three modules use a UMD boundary. They expose browser globals and can also be loaded with CommonJS from Node-based CI.

## Required operations

A conforming provider implements Promise-returning methods:

```text
listRecords(options?)
getRecord(recordId, options?)
upsertRecord(record, options?)
archiveRecord(recordId, options?)
restoreRecord(recordId, options?)
deleteRecord(recordId, options?)
exportWorkspace(options?)
healthCheck()
```

### listRecords

Returns active records by default. `includeArchived: true` may include Archive Vault records. Providers must return cloned or otherwise isolated values so callers cannot silently mutate provider state.

### getRecord

Returns `null` when no record exists. A found result contains:

- `record`
- `archived`
- `revisions`

Unknown record fields must survive round trips.

### upsertRecord

Creates a new record or updates an active record. An update requires the current `expectedConflictToken`. Omitting or supplying a stale token must produce a non-retryable `CONFLICT` error.

An archived record cannot be overwritten by upsert. It must be restored first.

### archiveRecord and restoreRecord

Archive moves the active record into the provider’s Archive Vault without deleting revision history. Restore moves it back to the active collection. A provider must not permit an active and archived copy with the same ID.

### deleteRecord

Permanent deletion requires explicit `{ permanent: true }`. Hosted implementations may add stronger server-side disposition requirements, but they must not weaken the client’s preservation, retention, approval, or disposition controls.

### exportWorkspace

Returns a `methodz-hosted-provider-export` package containing:

- active records
- archived records
- revision history
- provider metadata
- package integrity metadata

Private JWK material and recognized private-key fields are rejected before export. Existing record integrity, recovery, signature, redaction, approval, retention, and unknown extension fields must be preserved.

### healthCheck

Returns provider identity, contract version, capabilities, collection counts, and a timestamp. A successful health check does not prove authentication, authorization, tenant isolation, durable delivery, or legal compliance.

## Deterministic conflicts

Every stored record receives provider metadata:

```json
{
  "providerMetadata": {
    "contractVersion": "1.0.0",
    "version": 2,
    "conflictToken": "mhp:2:..."
  }
}
```

The token is derived from canonical JSON and the provider version. It is a deterministic concurrency token, not a cryptographic signature and not identity proof.

A future hosted provider may use an ETag, row version, transaction ID, or stronger digest internally, but it must present equivalent expected-token behavior to the client.

## Idempotency

`upsertRecord` accepts an optional `idempotencyKey` of up to 200 characters.

- Repeating the same key with the same request returns the original result and sets `idempotentReplay: true`.
- Reusing the key with different input produces `IDEMPOTENCY_CONFLICT`.
- Providers must scope keys so one tenant cannot collide with another tenant’s operations.

The local reference adapter stores idempotency metadata in `methodzProviderIdempotency`.

## Error contract

Provider failures use `ProviderError` with:

```text
code
retryable
operation
providerId
details
```

Retryable categories are:

- `PARTIAL_FAILURE`
- `UNAVAILABLE`
- `RATE_LIMITED`

Conflict, authorization, validation, integrity, and private-key rejection failures are non-retryable until the caller changes the request or authority context.

A remote provider must not hide partial success. It must identify completed and failed work in safe metadata without logging meeting content or private key material.

## Attachments

The v1.6.3 contract supports **attachment references only**. Existing attachment metadata may be preserved inside records.

Binary file transfer is outside this contract. A future binary provider requires a separate capability and threat review covering size limits, malware handling, content type, encryption, retention, deletion, and signed download behavior.

Base64 blobs, `data:` URLs, private keys, access tokens, and credentials must not be introduced into record-provider payloads.

## Revision, archive, retention, and recovery invariants

A provider must preserve:

- record schema and unknown fields
- Archive Vault state
- revision snapshots
- retention metadata
- preservation holds and event history
- disposition requests and audit metadata
- release approvals and receipts
- recipient policies and redaction metadata
- signature metadata and public keys
- workspace recovery-package integrity metadata

A hosted provider must never reinterpret provider synchronization as approval, signer identity, recipient identity, delivery proof, or authority.

## Security responsibilities outside this client contract

Production hosted providers still require server-side design and evidence for:

- authentication
- authorization
- tenant and organization isolation
- encryption in transit and at rest
- credential lifecycle
- server-side input validation
- append-only or tamper-evident audit storage
- rate limiting and abuse controls
- durable backups and disaster recovery
- data residency and retention enforcement
- observability without sensitive-data leakage
- incident response

Passing the client conformance suite does not certify any of these controls.

## Reference providers

### Disposable in-memory provider

Used only for tests and architecture exercises. It has no persistence and accepts injected synthetic failures.

### Browser localStorage provider

Uses disposable Storage-compatible objects during CI. In the app, it can bind to the established Methodz storage keys. The conformance suite never writes to production browser-local records.

## Compatibility rules

- App shell: `1.6.3`
- Record schema: `1.6.0`
- Hosted-provider contract: `1.0.0`
- No migration is required.
- Direct-file use remains supported.
- No runtime npm dependency is introduced.
- Script order is part of the static app contract.
