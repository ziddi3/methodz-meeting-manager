# v1.5 Cryptographic Package Architecture

## Components

```text
config-v15.js
  schema and key-registry configuration

migrations-v15.js
  record-level package-signature metadata

crypto-package-core.js
  portable canonicalization, private-material detection, key import/export,
  signing, and verification

features-v15-crypto.js
  meeting-workspace key management and package workflows

features-v15-key-safety.js
  public-key normalization, registry sanitation, and safe public exports

features-v15-record-metadata.js
  final record signature-control metadata and release-audit refresh

verify.html + verify.js
  standalone verifier with no meeting-record import
```

## Signing boundary

Signing is package-level and optional. It does not replace the v1.2 approval gate, v1.4 recipient policy, redaction, retention, legal-hold, or disposition controls.

Recommended flow:

```text
controlled source record
  -> redacted external package
  -> recipient policy
  -> release approval
  -> approved JSON download
  -> optional cryptographic signature
  -> independent verification
```

## Canonicalization

The signer removes any existing `signatureEnvelope`, sorts object keys recursively, preserves array order, and serializes the resulting JSON with `methodz-canonical-json-v1`.

The package content receives a SHA-256 payload digest. The ECDSA signature is calculated over a second canonical structure containing:

```text
package
signatureMetadata.version
signatureMetadata.type
signatureMetadata.signedAt
signatureMetadata.signerLabel
signatureMetadata.keyLabel
signatureMetadata.keyId
signatureMetadata.algorithm
signatureMetadata.payloadDigest
signatureMetadata.publicKeyJwk
signatureMetadata.notice
```

This binds both package content and displayed signature metadata into the signature. Changing the signer label, key label, signed timestamp, algorithm declaration, digest, key ID, public key, notice, or package content invalidates verification.

## Signature envelope

```text
signatureEnvelope.version
signatureEnvelope.type
signatureEnvelope.signedAt
signatureEnvelope.signerLabel
signatureEnvelope.keyLabel
signatureEnvelope.keyId
signatureEnvelope.algorithm
signatureEnvelope.payloadDigest
signatureEnvelope.signature
signatureEnvelope.publicKeyJwk
signatureEnvelope.notice
```

The public key is embedded so a package can be verified offline. The key ID is derived from the canonical P-256 public coordinates.

## Private-key handling

Private keys are deliberately excluded from browser storage, workspace backups, public-key exports, signed packages, and verification reports.

The page keeps the private JWK only in JavaScript memory. Refreshing or closing the page clears it. The explicit private-key backup download is the only built-in persistence path.

The cryptographic core rejects attempts to sign a package containing an EC private JWK or recognized private-key container. The public-key safety layer also normalizes imported keys to the P-256 public coordinates before registration, export, display, or reuse. Existing malformed registry entries are sanitized or discarded when the workspace loads.

## Record metadata

`externalSignatureControl` records the last signed-package timestamp, key ID, signature algorithm, and local verification timestamp for the source record. It does not store the signature body or private key.

The final v1.5 collection wrapper recomputes the release audit after signature-control metadata is attached so schema validation represents the complete saved record.

## Trust model

The signature establishes package and signature-metadata integrity relative to a key. Human identity and authority remain outside the browser-local trust boundary.

Future hosted architecture should add:

- authenticated users and roles;
- organization-managed keys;
- server-side signing or hardware-backed keys;
- durable revocation;
- append-only signature audit events;
- independent public-key distribution.
