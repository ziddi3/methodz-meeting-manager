# v1.5 Architecture: Policy Operations and Release Receipts

## Module order

```text
config-v14.js
config-v15.js
migrations-v14.js
migrations-v15.js
features-v14-recipient-policy.js
features-v14-policy-hardening.js
features-v15-policy-operations.js
```

Script order is part of the runtime contract. The v1.5 module wraps the final v1.4 preview and approved-download functions.

## Storage

```text
methodzRecipientPolicyGovernance
methodzExternalReleaseReceipts
```

Workspace backup already captures `methodz*` keys, so both collections are included without a special backup path.

## Governance model

Governance metadata is stored separately from the recipient policy record and joined by `policyId`.

```text
policyId
stewardName
stewardRole
riskTier
businessPurpose
cadenceDays
lastReviewedAt
lastReviewedBy
reviewNote
createdAt
updatedAt
```

Keeping stewardship separate avoids changing the v1.4 policy normalization contract. Marking a policy reviewed updates the v1.4 `reviewDate`, so existing overdue-policy enforcement continues to work.

## Fingerprint binding

When a recipient policy has governance metadata, the preview adds:

```text
record.externalCopy.recipientGovernanceVersion
record.externalCopy.recipientRiskTier
manifest.recipientGovernance
```

The governance version is part of the redacted record used by the external approval fingerprint. Changing governance metadata therefore requires a new matching approval.

## Release receipt ledger

Every successful approved download produces a receipt containing:

- approval ID and approval snapshot;
- source record reference;
- destination and recipient policy snapshot;
- governance snapshot;
- redaction profile;
- package integrity value;
- release format and timestamp;
- previous receipt digest and current receipt digest.

The ledger uses deterministic canonical JSON and FNV-1a-32 for local chain verification. This is a compatibility checksum and local change-detection aid, not cryptographic signing.

When the bounded ledger is trimmed, retained receipts are re-chained from a new local genesis so verification remains internally consistent.

## Source record metadata

The latest receipt updates:

```text
externalRecipientControl.lastReleaseReceiptId
externalRecipientControl.lastReleaseReceiptAt
externalRecipientControl.lastReleaseIntegrityAlgorithm
externalRecipientControl.lastReleaseIntegrityDigest
```

Active and archived records are both supported.

## Future provider boundary

A hosted provider should replace the local receipt writer with an authenticated append-only service. Server receipts should use durable identities, trusted timestamps, authorization checks, and managed cryptographic signing only when key custody is explicitly designed.
