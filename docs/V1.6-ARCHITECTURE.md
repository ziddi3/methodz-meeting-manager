# v1.6 Cryptographic Package Architecture

## Components

```text
config-v16.js
  schema, signing-key storage keys, and protocol configuration

migrations-v16.js
  record-level externalSignatureControl migration and validation

crypto-package-core.js
  canonicalization, private-material detection, JWK handling,
  ECDSA signing, and verification

features-v15-crypto.js
  signing workspace and key-management interface

features-v15-key-safety.js
  public-key normalization, registry sanitation, and safe public exports

features-v16-crypto-release.js
  v1.6 release API aliases and visible release labeling

features-v16-record-metadata.js
  final saved-record signature metadata and schema-audit refresh

verify.html + verify.js
  standalone verifier with no meeting-record import
```

The cryptographic core and workspace modules retain their protocol-level V15 identifiers for compatibility with the completed implementation branch. `features-v16-crypto-release.js` exposes stable V16 aliases for release consumers.

## Signing boundary

Signing is package-level and optional. It does not replace recipient policy, release approval, release receipts, redaction, retention, legal-hold, or disposition controls.

Recommended flow:

```text
controlled source record
  -> redacted external package
  -> recipient allow-list
  -> release approval
  -> approved JSON package
  -> optional ECDSA signature
  -> independent verification
```

## Canonicalization

The signer removes any existing `signatureEnvelope`, sorts object keys recursively, preserves array order, and serializes the result using `methodz-canonical-json-v1`.

The package receives a SHA-256 payload digest. The ECDSA signature covers a second canonical structure containing both the package and displayed signature metadata:

```text
package
signatureMetadata.version
signatureMetadata.type
signatureMetadata.signedAt
signatureMetadata.signerLabel
signatureMetadata.keyLabel
signatureMetadata.keyId
signatureMetadata.algorithm
signatureMetadata.payloadDigest
signatureMetadata.publicKeyJwk
signatureMetadata.notice
```

Changing the package, signer label, key label, signing timestamp, algorithm declaration, digest, key ID, public key, or trust notice invalidates verification.

## Private-key handling

Private keys are excluded from browser storage, workspace backups, public-key exports, signed packages, service-worker caches, and verification reports.

The page keeps the private JWK only in JavaScript memory. Refreshing or closing the page clears it. The explicit sensitive backup download is the only built-in persistence path.

The core rejects packages containing an EC private JWK or recognized private-key container. Public-key imports are normalized to the P-256 public coordinates before registration or export. Existing malformed registry entries are sanitized or discarded when the workspace loads.

## Record metadata

`externalSignatureControl` stores:

```text
optional
lastSignedPackageAt
lastSigningKeyId
lastSignatureAlgorithm
lastVerificationAt
```

It does not store a private key or signature body. The final v1.6 collection wrapper recomputes the release audit after signature metadata is attached.

## Trust model

The signature establishes package and signature-metadata integrity relative to a key. Human identity and authority remain outside the browser-local trust boundary.

A future hosted provider should add authenticated users, organization-managed keys, hardware-backed or server-side signing, durable revocation, append-only signature audits, and independently distributed public keys.
