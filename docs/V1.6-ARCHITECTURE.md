# v1.6 Cryptographic Package Architecture

## Purpose

Version 1.6 adds optional, portable cryptographic signatures to exported JSON packages without changing the offline-first, static deployment model. It preserves all v1.5 recipient-policy, approval, receipt, retention, preservation, and disposition controls.

## Components

```text
config-v16.js
  schema version, signing storage keys, protocol limits, and supported package types

migrations-v16.js
  externalSignatureControl migration and validation

crypto-package-core.js
  canonical JSON, private-material detection, JWK handling,
  ECDSA P-256 signing, SHA-256 digests, and verification

features-v16-crypto.js
  signing workspace, public-key registry, key lifecycle, audit exports,
  package loading, signing, verification, and verification-report downloads

features-v16-record-metadata.js
  final saved-record signature metadata and schema-audit refresh

verify.html + verify.js
  standalone verifier that does not import a package into meeting storage
```

## Signing boundary

Signing is package-level and optional. It occurs after an approved JSON package has been produced.

```text
controlled source record
  -> redaction profile
  -> recipient allow-list
  -> governance-version binding
  -> content fingerprint
  -> destination-bound approval
  -> approved JSON package and release receipt
  -> optional ECDSA signature
  -> independent verification
```

A signature does not replace any earlier control in that pipeline.

## Canonicalization

The signer removes any existing `signatureEnvelope`, sorts object keys recursively, preserves array order, and serializes the result using `methodz-canonical-json-v1`.

The package receives a SHA-256 payload digest. The ECDSA signature covers a second canonical object containing both the unsigned package and displayed signature metadata:

```text
signatureMetadata.version
signatureMetadata.type
signatureMetadata.signedAt
signatureMetadata.signerLabel
signatureMetadata.keyLabel
signatureMetadata.keyId
signatureMetadata.algorithm
signatureMetadata.payloadDigest
signatureMetadata.publicKeyJwk
signatureMetadata.notice
```

Changing package content, signer label, key label, timestamp, algorithm declaration, digest, key ID, public key, or trust notice invalidates verification.

## Key identity

The key ID is derived from SHA-256 over the canonical P-256 public coordinates and is represented as:

```text
p256:<first-32-hex-characters>
```

This gives the application a stable public-key reference. It does not prove who controls the corresponding private key. Key IDs should be confirmed through an independent trusted channel.

## Private-key handling

Private keys are excluded from:

- browser storage;
- workspace backups;
- public-key registry exports;
- signed packages;
- verification reports;
- service-worker cache entries;
- saved meeting records.

The page keeps imported or generated private JWK material only in JavaScript memory. Refreshing or closing the page clears it. The explicit sensitive backup download is the only built-in persistence path.

The cryptographic core rejects packages containing an EC private JWK or recognized private-key container. Public-key imports are normalized to public P-256 coordinates before registration or export. Malformed or unsafe registry entries are removed when the workspace initializes.

## Public-key registry

`methodzSigningPublicKeys` stores public verification keys and browser-local workflow metadata:

```text
id
keyLabel
signerLabel
status
publicKeyJwk
source
createdAt
updatedAt
revokedAt
```

`Active` and `Revoked` states help local workflow decisions. They are not authenticated, organization-wide, or durable revocation infrastructure.

## Record metadata

Current records carry:

```text
externalSignatureControl.optional
externalSignatureControl.lastSignedPackageAt
externalSignatureControl.lastSigningKeyId
externalSignatureControl.lastSignatureAlgorithm
externalSignatureControl.lastVerificationAt
```

The record does not store a private key or signature body. Package signing updates source metadata only when a source record ID can be resolved.

## Audit boundary

`methodzSigningAudit` records browser-local events such as generation, import, clearing, signing, verification, revocation, restoration, and sanitation. Audit exports are operational evidence, not immutable authenticated logs.

## Static deployment

All modules are ordinary static files. The main workspace, archive view, verifier, manifest, and service worker can be deployed without a build process or runtime package dependency.

Web Crypto is normally available on HTTPS and localhost. Direct-file meeting workflows remain supported, but signing availability may vary by browser security context.

## Hosted-provider future

A hosted 2.0 provider should add:

- authenticated users and role enforcement;
- organization-managed public keys;
- controlled key issuance and rotation;
- durable revocation records;
- hardware-backed or server-side signing where appropriate;
- append-only remote signature and release audits;
- independent public-key distribution;
- provider conformance tests against the existing adapter boundaries.
