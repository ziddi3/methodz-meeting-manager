# Methodz Meeting Manager v1.6.2 Verification Conformance

## Purpose

This document defines the portable verification checks used to prove that a Methodz ECDSA P-256 / SHA-256 package can be generated without retaining private key material and verified consistently by the standalone verifier in supported browser engines.

The conformance layer does not change the meeting-record schema. It strengthens release validation around the existing v1.6 cryptographic package protocol and the v1.6.2 public-key custody workflow.

## Disposable fixture model

The test suite creates a synthetic signed package at runtime using an ephemeral P-256 key pair.

The private key:

- exists only inside the short-lived Node process;
- is never printed;
- is never written to disk;
- is never added to browser storage;
- is never uploaded as a workflow artifact;
- is discarded when the generator process exits.

The generated JSON fixture contains only:

- synthetic public meeting data;
- the public P-256 verification key;
- the derived Methodz key ID;
- the payload digest;
- signature metadata;
- the ECDSA signature.

The fixture is explicitly non-production and must not be used as evidence of human identity, authority, approval, delivery, or legal validity.

## Automated checks

### Node verification

`tests/v162-fixture-core.mjs` verifies that:

- the generated fixture contains no private JWK material;
- the embedded public key contains no `d` coordinate;
- the package signature is valid;
- the payload digest matches;
- the derived public-key ID matches;
- payload tampering is rejected;
- signature-metadata tampering is rejected;
- injected private material is detected.

### Browser verification

`tests/v162-fixture.spec.js` drives the standalone `verify.html` interface and checks that:

- the generated package loads through the file input;
- the package summary is rendered;
- verification reports `Valid Signature` for the untouched fixture;
- the signer and key labels remain visible;
- a payload-tampered copy reports `Invalid Signature`.

The focused fixture suite runs in:

- Chromium;
- Firefox;
- WebKit.

The larger historical regression suites remain Chromium-based to keep workflow duration controlled while the portable cryptographic boundary receives multi-engine coverage.

## Workflow artifacts

The validation job uploads the public disposable fixture as a short-retention workflow artifact. This allows a failed run to be inspected without exposing a private key.

Browser traces are uploaded only when a browser suite fails. Artifact names include both the suite and browser engine so failures remain isolated.

## Local validation

From the repository root:

```bash
node tests/generate-v162-disposable-fixture.mjs
node tests/v162-fixture-core.mjs
```

To run a focused browser check after installing Playwright:

```bash
METHODZ_BROWSER=chromium npx playwright test tests/v162-fixture.spec.js
METHODZ_BROWSER=firefox npx playwright test tests/v162-fixture.spec.js
METHODZ_BROWSER=webkit npx playwright test tests/v162-fixture.spec.js
```

## Deployment boundary

These tests add no runtime dependency to the deployed application. The production app remains plain static HTML, CSS, and JavaScript. Direct-file meeting workflows continue to function without a server, package manager, or build command.

The standalone verifier still requires Web Crypto support. On browsers that restrict Web Crypto under `file://`, verification should be run from HTTPS or localhost.
